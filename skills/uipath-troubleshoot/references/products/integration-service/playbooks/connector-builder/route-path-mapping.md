# Custom Connector Route and Path Mapping

## Context

Signature: a custom connector works in Builder preview or another environment, but the
published runtime returns `404`, often with text similar to “No API found at that URL for the
given token and credentials.”

Do not classify this as bad credentials from the message alone. First determine whether the
`404` was generated by Integration Service route matching or by the provider.

Required anchors: environment, connector key and published version, connection ID,
resource/operation and HTTP method, runtime URL suffix, request/trace ID and UTC timestamp,
and whether any request reached the provider.

### What can cause it

- The published exposed Integration Service path or HTTP method does not match the generated activity route.
- The provider `vendorPath`, base URL, API version, resource identifier, or HTTP method is wrong.
- The local definition differs from the imported/published revision used by the runtime.
- An authoring/import capability mismatch or save operation copied the provider path into the exposed route.
- Provider authorization masks an inaccessible resource as `404` after the request reaches the provider.

## Investigation

1. **Identify the `404` producer.**
   - Provider request/response or provider request ID exists → follow the provider path branch.
   - No provider request and IS emits a route/no-API message → investigate the exposed route.
2. **Inspect and validate the source definition:**

   ```bash
   uip is connectors builder inspect --output json
   uip is connectors builder validate --output json
   ```

3. **Compare all four values for the failing method:**
   - the public/exposed resource path used by the Studio activity;
   - the provider `vendorPath`;
   - the HTTP method;
   - the actual runtime URL suffix.
4. **Compare revisions.** Verify that the definition inspected locally is the one imported and
   published in the affected environment. Builder preview and a different tenant/environment
   are not proof of the published route.
5. **Check edit history.** Determine whether an imported operation was opened and saved before
   the failure. A known failure family can copy the provider path into the exposed path when
   hooks are enabled but path-mapping capability is not consistently enabled in that
   environment.
6. **If the route matches and the request reached the provider,** inspect base URL, provider
   API version, path parameters, URL encoding, and provider permissions. Route this provider
   `404` to [request-failed.md](../request-failed.md).

### Diagnosis

| Evidence | Diagnosis |
|---|---|
| No provider request; exposed path contains the vendor path or differs from runtime route | Exposed-path/path-mapping defect |
| Local definition is correct but imported/published revision differs | Stale or wrong imported/published revision |
| Method differs between activity and published resource | Method/route contract mismatch |
| IS routes successfully and provider returns `404` | Provider path, version, identifier, or permission issue |
| Only one environment fails with the same definition and revision | Environment capability/deployment inconsistency; escalate |

## Resolution

Quote one confirmed cause verbatim from **What can cause it**, then correct only
that route/provider branch. If evidence does not isolate one cause, stop at the
missing discriminator.

Diagnosis is read-only. Obtain explicit approval before editing a route, importing,
publishing, recreating a connection, or rebinding a consumer.

- For an exposed-path defect, preserve the public resource path separately from `vendorPath`,
  validate, import, and republish the corrected definition.
- If environment capability state caused import/save corruption, have the owning team correct
  that state before a clean re-import and republish. Do not repeatedly save the resource while
  the mismatch remains.
- Recreate or rebind the connection only if the corrected publish changed connector identity or
  the existing connection still references the bad published definition.
- For a provider `404`, correct the provider base URL/version/path/identifier or access issue;
  do not alter the exposed route.

### Verification

Confirm the intended exposed path, `vendorPath`, method, imported version, and published version
all agree. Then use a fresh or correctly rebound connection to prove the same operation reaches
the provider in the affected environment.

### Escalation Bundle

Include anchors, redacted local and exported/published resource entries, expected versus actual
route and method, publish ID/version, environment comparison, provider-reachability evidence,
and whether the operation was edited after import.
