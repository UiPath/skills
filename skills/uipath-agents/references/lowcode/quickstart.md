# UiPath Low-Code Agent Authoring ŌĆö Quickstart

## When to Use

- Create a new low-code agent project (standalone or inline in a flow)
- Edit `agent.json` ŌĆö prompts, model, schemas, settings, resources
- Add tools to an agent: RPA processes, other agents, Integration Service connections
- Add RAG contexts (semantic index) and escalations (human-in-the-loop)
- Wire agent-to-agent calls within a solution or to an external deployed agent
- Design input/output schemas and sync with `entry-points.json`
- Validate agent project structure
- Publish agent to Studio Web, pack and deploy to Orchestrator

## Critical Rules

1. **Edit JSON files directly** ŌĆö the CLI supports `init` (scaffold) and `validate` (check + migrate). Agent configuration (prompts, schemas, settings) is done by editing `agent.json` and `entry-points.json`. Resources (tools, contexts, escalations) are added as individual files in `resources/{ResourceName}/resource.json` inside the agent project directory ŌĆö **not** inline in `agent.json`. The root `agent.json` should not contain a `resources` field. The `validate` command reads these resource files and generates `.agent-builder/agent.json` which inlines them.

2. **Validate after every bulk of edits** ŌĆö run `uip agent validate --output json` after completing a set of related changes. Validate: (1) statically checks `agent.json` structure, `contentTokens` consistency, `entry-points.json` schema sync, and `project.uiproj`; (2) applies all pending migrations in memory and returns errors without touching disk if migration fails; (3) post-migration validates `agent.json` schema, eval-sets, and evaluators; (4) writes migrated files back to disk only when all checks pass.

3. **Use `--output json`** on all `uip` commands when parsing output.

4. **Keep schemas in sync** ŌĆö `inputSchema` and `outputSchema` in `agent.json` must exactly mirror `input` and `output` in `entry-points.json`. Update both when adding or removing fields.

5. **Use `{{input.fieldName}}` syntax** in user message templates. Do not use `$vars` or `=js:` expressions ŌĆö those are flow syntax, not agent syntax.

6. **Keep contentTokens in sync with content** ŌĆö every message has both a `content` string and a `contentTokens` array. Update both when editing. See [agent-json-format.md](agent-json-format.md) ┬¦ contentTokens Construction.

7. **Do not manually create or edit `.agent-builder/`** ŌĆö it is generated by `uip agent validate` and managed by Studio Web. Never create or modify `.agent-builder/` files by hand.

8. **Do not publish or deploy without user consent** ŌĆö ask before running `uip solution upload`, `uip solution publish`, or `uip solution deploy`.

9. **Do not modify `projectId`** ŌĆö auto-generated by `uip agent init`.

10. **Solution must exist first** ŌĆö create one with `uip solution new` before scaffolding an agent.

11. **Use `"folderPath": "solution_folder"` for resources internal to the solution.** This placeholder resolves to the actual deployment folder at runtime. External resources use their real Orchestrator folder path (e.g., `"Shared"`, `"Shared/TestRPA"`).

12. **Agent tools have `"location": "solution"` or `"location": "external"`.** Solution agents use `"location": "solution"` and `"folderPath": "solution_folder"`. External agents use `"location": "external"` and their actual folder path.

13. **Never manually edit `storageVersion`.** It is managed by `uip agent validate` (which migrates files to the latest version on success) and by Studio Web on import. If validate reports that `storageVersion` is newer than supported, upgrade uipcli rather than editing the field by hand.

14. **Inline agents in flows use `uipath.agent.autonomous` nodes.** The node's `model.source` references the inline agent's `projectId` UUID. The agent definition lives in a subdirectory inside the flow project. See [agent-flow-integration.md](agent-flow-integration.md).

15. **Never invoke other skills automatically.** If the user needs flow operations, tell them to use the `uipath-maestro-flow` skill.

## Common Edits

### Change System Prompt

1. Edit `agent.json` ŌåÆ `messages[0].content`
2. Rebuild `messages[0].contentTokens` ŌĆö single `simpleText` entry for prompts with no variables
3. Validate: `uip agent validate --output json`

### Change User Message Template

1. Edit `agent.json` ŌåÆ `messages[1].content`
2. Rebuild `messages[1].contentTokens` ŌĆö tokenize `{{input.fieldName}}` as `variable`, surrounding text as `simpleText`
3. Validate

### Add an Input Field

1. Add to `agent.json` ŌåÆ `inputSchema.properties` (and `.required` if mandatory)
2. Mirror in `entry-points.json` ŌåÆ `entryPoints[0].input.properties` (and `.required`)
3. Update `messages[1].content` and `contentTokens` if the field should appear in the user message
4. Validate

### Add an Output Field

1. Add to `agent.json` ŌåÆ `outputSchema.properties`
2. Mirror in `entry-points.json` ŌåÆ `entryPoints[0].output.properties`
3. Validate

### Add a Tool (RPA process, agent, Integration Service)

1. Create `resources/{ToolName}/resource.json` inside the agent project directory with `$resourceType: "tool"` ŌĆö see [agent-json-format.md](agent-json-format.md) ┬¦ Resources
2. For solution-internal tools: set `"location": "solution"`, `"folderPath": "solution_folder"`, and `"referenceKey": ""` (validate resolves it and writes it back to disk)
3. For external tools: set `"location": "external"` and actual `"folderPath"`
4. Validate ŌĆö this generates `.agent-builder/` files and resolves `referenceKey` for solution tools

### Change Model Settings

1. Edit `agent.json` ŌåÆ `settings.model`, `.temperature`, `.maxTokens`, or `.maxIterations`
2. Current models: `anthropic.claude-sonnet-4-6`, `gpt-4.1-2025-04-14`, `gpt-5.2-2025-12-11`
3. Validate

## Quick Start: Scenario 1 ŌĆö New Standalone Agent

### Step 0 ŌĆö Resolve `uip` binary

```bash
which uip || npm root -g 2>/dev/null | xargs -I{} echo {}/uip/bin/uip
```

If not found: `npm install -g @uipath/uip`

### Step 1 ŌĆö Check login status

```bash
uip login status --output json
```

If not logged in, prompt the user to run `uip login`.

### Step 2 ŌĆö Create solution and scaffold agent

```bash
uip solution new "<SOLUTION_NAME>" --output json
cd "<SOLUTION_NAME>"
uip agent init "<AGENT_NAME>" --output json
uip solution project add "<AGENT_NAME>" --output json
```

### Step 3 ŌĆö Configure agent.json

Read [agent-json-format.md](agent-json-format.md) for the full schema.

1. Set `settings.model` (e.g., `"anthropic.claude-sonnet-4-6"`)
2. Set `settings.temperature` (0 for deterministic)
3. Write system prompt in `messages[0].content` + rebuild `contentTokens`
4. Write user message template in `messages[1].content` using `{{input.fieldName}}` + rebuild `contentTokens`

### Step 4 ŌĆö Define input/output schemas

1. Add fields to `agent.json` ŌåÆ `inputSchema` and `outputSchema`
2. Mirror in `entry-points.json`
3. Validate: `uip agent validate "<AGENT_NAME>" --output json`

### Step 5 ŌĆö Publish to Studio Web or deploy to Orchestrator

Ask the user before proceeding. There are two separate paths:

**Studio Web** (default ŌĆö for visual editing and sharing):
```bash
uip solution upload . --output json
```

**Orchestrator** (for production deployment ŌĆö only when explicitly requested):
```bash
uip solution pack . ./dist -v "1.0.0" --output json
uip solution publish ./dist/<SOLUTION_NAME>.1.0.0.zip --output json
uip solution deploy run --name "<NAME>" --package-name "<SOLUTION_NAME>" --package-version "1.0.0" --output json
```

## Quick Start: Scenario 2 ŌĆö Multi-Agent Solution (Parent + Tool Agents)

Use when one agent orchestrates others via tool resources in the `resources/` directory.

```bash
# 1. Create solution
uip solution new "<SOLUTION_NAME>" --output json
cd "<SOLUTION_NAME>"

# 2. Scaffold both agents
uip agent init "ParentAgent" --output json
uip agent init "ToolAgent" --output json

# 3. Link both to the solution
uip solution project add "ParentAgent" --output json
uip solution project add "ToolAgent" --output json

# 4. Create ParentAgent/resources/ToolAgent/resource.json with the tool definition
# See agent-json-format.md ┬¦ Resources for the full format.
# Use location: "solution" + folderPath: "solution_folder" for solution-internal tools.
# Use location: "external" + actual folderPath for external agent tools.

# 5. Validate both (generates .agent-builder/ files and resolves referenceKey)
uip agent validate ParentAgent --output json
uip agent validate ToolAgent --output json

```

See [agent-solution-guide.md](agent-solution-guide.md) for resource file formats and UUID cross-references.

## Quick Start: Scenario 3 ŌĆö Inline Agent in a Flow

Use when embedding an agent directly inside a flow project. The agent lives as a subdirectory and is published with the flow.

```bash
# 1. Ensure solution and flow project exist
# (use uipath-maestro-flow skill to create them, or start from existing)

# 2. Scaffold the inline agent inside the flow project
uip agent init "<FlowProjectDir>" --inline-in-flow --output json
# Returns the generated projectId (UUID) and path

# 3. Edit the agent.json inside <FlowProjectDir>/<projectId>/
# - Set system prompt in messages[0].content + rebuild contentTokens
# - Set model in settings.model
# - Configure outputSchema if needed

# 4. Add tools to <FlowProjectDir>/<projectId>/resources/ (optional)
# Same resource.json format as standalone agents

# 5. Validate the inline agent
uip agent validate "<FlowProjectDir>/<projectId>" --inline-in-flow --output json

# 6. Add the inline agent node to the flow
uip flow node add <FlowName>.flow uipath.agent.autonomous \
  --source <projectId> \
  --label "Autonomous Agent" \
  --output json

# 7. Wire edges to connect the agent node
uip flow edge add <FlowName>.flow <sourceNodeId> <agentNodeId> \
  --source-port success \
  --target-port input \
  --output json
```

See [embedding-in-flows.md](embedding-in-flows.md) for the full structure and [agent-flow-integration.md](agent-flow-integration.md) for the flow node format.

## Inline Agents in Flow Projects

Agents can be embedded as a subdirectory inside a flow project. Read [embedding-in-flows.md](embedding-in-flows.md) for the full guide.

Key differences from standalone:
- Folder name is the agent's `projectId` UUID (not a human-readable name)
- Contains `agent.json`, `flow-layout.json` (empty `{}`), `evals/eval-sets/` (empty), `features/`, `resources/`
- No `entry-points.json`, no `project.uiproj` ŌĆö governed by the parent flow project
- Published with the parent flow
- Flow wiring is handled by the `uipath-maestro-flow` skill

## What NOT to Do

1. **Do not manually create or edit `.agent-builder/` files** ŌĆö it is generated by `uip agent validate` and managed by Studio Web. Never create or modify these files by hand.
2. **Do not use `=js:` or `$vars` in agent messages** ŌĆö use `{{input.fieldName}}` only
3. **Do not skip schema sync** ŌĆö agent.json and entry-points.json must match
4. **Do not batch edits before validating** ŌĆö validate after each change
5. **Do not publish/deploy without validating** ŌĆö always validate first
6. **Do not forget contentTokens** ŌĆö editing `content` without updating `contentTokens` causes rendering issues
7. **Do not use `folderPath: "solution_folder"` for truly external resources** ŌĆö use the actual Orchestrator folder path
8. **Do not copy-paste UUIDs from one resource to another** ŌĆö every resource needs a unique UUID
9. **Do not bump `storageVersion` manually** ŌĆö breaks packager compatibility
10. **Do not call raw Automation.Solutions REST APIs** ŌĆö always use `uip solution` commands

## Completion Output

After completing a task, report:

1. **File paths** ŌĆö which files were created or modified
2. **What was configured** ŌĆö summary of agent settings and schemas
3. **Validation result** ŌĆö output of `uip agent validate --output json`; confirm `MigrationApplied`, `StorageVersion`, and `Validated` counts
4. **Schema sync status** ŌĆö confirm agent.json and entry-points.json match
