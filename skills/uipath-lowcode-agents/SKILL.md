# UiPath Low-Code Agent Authoring Assistant

## When to Use This Skill

- Create a new low-code agent project (standalone or inline in a flow)
- Edit `agent.json` ‚Äî prompts, model, schemas, settings, resources
- Add tools to an agent: RPA processes, other agents, Integration Service connections
- Add RAG contexts (semantic index) and escalations (human-in-the-loop)
- Wire agent-to-agent calls within a solution or to an external deployed agent
- Design input/output schemas and sync with `entry-points.json`
- Validate agent project structure
- Publish agent to Studio Web, pack and deploy to Orchestrator

## Critical Rules

1. **Edit JSON files directly** ‚Äî the CLI supports `init` (scaffold) and `validate` (check). All configuration is done by editing `agent.json` and `entry-points.json`. Resources (tools, contexts, escalations) are added by editing the `resources[]` array in `agent.json`.

2. **Validate after every change** ‚Äî run `uip lowcodeagents validate --output json` after each edit. Do not batch multiple edits before validating.

3. **Use `--output json`** on all `uip` commands when parsing output.

4. **Keep schemas in sync** ‚Äî `inputSchema` and `outputSchema` in `agent.json` must exactly mirror `input` and `output` in `entry-points.json`. Update both when adding or removing fields.

5. **Use `{{input.fieldName}}` syntax** in user message templates. Do not use `$vars` or `=js:` expressions ‚Äî those are flow syntax, not agent syntax.

6. **Keep contentTokens in sync with content** ‚Äî every message has both a `content` string and a `contentTokens` array. Update both when editing. See [references/agent-json-format.md](references/agent-json-format.md) ¬ß contentTokens Construction.

7. **Do not touch `.agent-builder/`** ‚Äî managed by Studio Web. Do not create, edit, or regenerate files inside it.

8. **Do not publish or deploy without user consent** ‚Äî ask before running `uip solution upload`, `uip solution publish`, or `uip solution deploy`.

9. **Do not modify `projectId`** ‚Äî auto-generated by `uip lowcodeagents init`.

10. **Solution must exist first** ‚Äî create one with `uip solution new` before scaffolding an agent.

11. **Use `"folderPath": "solution_folder"` for resources internal to the solution.** This placeholder resolves to the actual deployment folder at runtime. External resources use their real Orchestrator folder path (e.g., `"Shared"`, `"Shared/TestRPA"`).

12. **Agent tools have `"location": "solution"` or `"location": "external"`.** Solution agents use `"location": "solution"` and `"folderPath": "solution_folder"`. External agents use `"location": "external"` and their actual folder path.

13. **The `storageVersion` field in `agent.json` is never manually edited.** Managed by Studio Web and the packager. Current value is `50.0.0`.

14. **There are 5 agent-in-flow patterns.** `uipath.agent.autonomous` = inline (P1). `uipath.core.agent.<guid>` with `section: "In this solution"` = solution agent (P2). `uipath.core.agent.<guid>` with `section: "Published"` = external agent (P3). `uipath.agent.resource.tool.agent.<guid>` with `section: "Published"` = external agent tool (P4). `uipath.agent.resource.tool.agent.<guid>` with `section: "In this solution"` = solution agent tool (P5). P4 and P5 only connect to an inline agent's `tool` handle ‚Äî never to the main flow sequence.

15. **Never invoke other skills automatically.** If the user needs flow operations, tell them to use the `uipath-flow` skill.

## Common Edits

### Change System Prompt

1. Edit `agent.json` ‚Üí `messages[0].content`
2. Rebuild `messages[0].contentTokens` ‚Äî single `simpleText` entry for prompts with no variables
3. Validate: `uip lowcodeagents validate --output json`

### Change User Message Template

1. Edit `agent.json` ‚Üí `messages[1].content`
2. Rebuild `messages[1].contentTokens` ‚Äî tokenize `{{input.fieldName}}` as `variable`, surrounding text as `simpleText`
3. Validate

### Add an Input Field

1. Add to `agent.json` ‚Üí `inputSchema.properties` (and `.required` if mandatory)
2. Mirror in `entry-points.json` ‚Üí `entryPoints[0].input.properties` (and `.required`)
3. Update `messages[1].content` and `contentTokens` if the field should appear in the user message
4. Validate

### Add an Output Field

1. Add to `agent.json` ‚Üí `outputSchema.properties`
2. Mirror in `entry-points.json` ‚Üí `entryPoints[0].output.properties`
3. Validate

### Add a Tool (RPA process, agent, Integration Service)

1. Add an entry to `agent.json` ‚Üí `resources[]` with `$resourceType: "tool"` ‚Äî see [agent-json-format.md](references/agent-json-format.md) ¬ß Resources
2. Validate

### Change Model Settings

1. Edit `agent.json` ‚Üí `settings.model`, `.temperature`, `.maxTokens`, or `.maxIterations`
2. Current models: `anthropic.claude-sonnet-4-6`, `gpt-4.1-2025-04-14`, `gpt-5.2-2025-12-11`
3. Validate

## Quick Start: Scenario 1 ‚Äî New Standalone Agent

### Step 0 ‚Äî Resolve `uip` binary

```bash
which uip || npm root -g 2>/dev/null | xargs -I{} echo {}/uip/bin/uip
```

If not found: `npm install -g @uipath/uip`

### Step 1 ‚Äî Check login status

```bash
uip login status --output json
```

If not logged in, prompt the user to run `uip login`.

### Step 2 ‚Äî Create solution and scaffold agent

```bash
uip solution new "<SOLUTION_NAME>" --output json
cd "<SOLUTION_NAME>"
uip lowcodeagents init "<AGENT_NAME>" --output json
uip solution project add --project-path "<AGENT_NAME>" --output json
```

### Step 3 ‚Äî Configure agent.json

Read [references/agent-json-format.md](references/agent-json-format.md) for the full schema.

1. Set `settings.model` (e.g., `"anthropic.claude-sonnet-4-6"`)
2. Set `settings.temperature` (0 for deterministic)
3. Write system prompt in `messages[0].content` + rebuild `contentTokens`
4. Write user message template in `messages[1].content` using `{{input.fieldName}}` + rebuild `contentTokens`

### Step 4 ‚Äî Define input/output schemas

1. Add fields to `agent.json` ‚Üí `inputSchema` and `outputSchema`
2. Mirror in `entry-points.json`
3. Validate: `uip lowcodeagents validate --output json`

### Step 5 ‚Äî Publish to Studio Web or deploy to Orchestrator

Ask the user before proceeding.

```bash
# Upload to Studio Web (for visual development)
uip solution upload --output json

# Or pack + publish + deploy to Orchestrator
uip solution pack . ./output -v "1.0.0" --output json
uip solution publish ./output/<SOLUTION_NAME>.1.0.0.zip --output json
uip solution deploy run \
  --name "<SOLUTION_NAME>-v1" \
  --package-name "<SOLUTION_NAME>" \
  --package-version "1.0.0" \
  --folder-name "<SOLUTION_NAME>" \
  --folder-path "Shared" \
  --output json
```

## Quick Start: Scenario 2 ‚Äî Multi-Agent Solution (Parent + Tool Agents)

Use when one agent orchestrates others via the `resources[]` tool array.

```bash
# 1. Create solution
uip solution new "<SOLUTION_NAME>" --output json
cd "<SOLUTION_NAME>"

# 2. Scaffold both agents
uip lowcodeagents init "ParentAgent" --output json
uip lowcodeagents init "ToolAgent" --output json

# 3. Link both to the solution
uip solution project add --project-path "ParentAgent" --output json
uip solution project add --project-path "ToolAgent" --output json

# 4. Edit ParentAgent/agent.json ‚Äî add ToolAgent as a solution tool in resources[]
# See references/agent-json-format.md ¬ß Resources for the full format.
# For an external agent tool, use location: "external" and actual folderPath.

# 5. Validate both
cd ParentAgent && uip lowcodeagents validate --output json && cd ..
cd ToolAgent && uip lowcodeagents validate --output json && cd ..
```

See [references/agent-solution-guide.md](references/agent-solution-guide.md) for resource file formats and UUID cross-references.

## Quick Start: Scenario 3 ‚Äî Five Patterns for Agents in Flows

There are five distinct patterns for embedding agents in flows. See [references/agent-flow-integration.md](references/agent-flow-integration.md) for full detail with node structures and bindings.

| # | Pattern | Node type | `model.section` |
|---|---------|-----------|-----------------|
| 1 | Inline agent | `uipath.agent.autonomous` | *(not a service task)* |
| 2 | Solution agent | `uipath.core.agent.<key>` | `"In this solution"` |
| 3 | External agent | `uipath.core.agent.<key>` | `"Published"` |
| 4 | External agent tool | `uipath.agent.resource.tool.agent.<key>` | `"Published"` |
| 5 | Solution agent tool | `uipath.agent.resource.tool.agent.<key>` | `"In this solution"` |

P4 and P5 are never standalone nodes ‚Äî they connect to the inline agent's `tool` handle only.

For flow operations (creating the flow file, adding nodes, connecting edges), use the `uipath-flow` skill.

## Inline Agents in Flow Projects

Agents can be embedded as a subdirectory inside a flow project. Read [references/embedding-in-flows.md](references/embedding-in-flows.md) for the full guide.

Key differences from standalone:
- No `entry-points.json`, no `flow-layout.json`, no `project.uiproj` ‚Äî only `agent.json` (plus `features/` and `resources/` subdirectories)
- Governed by the parent flow project
- Published with the parent flow
- Flow wiring is handled by the `uipath-flow` skill

## What NOT to Do

1. **Do not edit `.agent-builder/` files** ‚Äî managed by Studio Web
2. **Do not use `=js:` or `$vars` in agent messages** ‚Äî use `{{input.fieldName}}` only
3. **Do not skip schema sync** ‚Äî agent.json and entry-points.json must match
4. **Do not batch edits before validating** ‚Äî validate after each change
5. **Do not publish/deploy without validating** ‚Äî always validate first
6. **Do not forget contentTokens** ‚Äî editing `content` without updating `contentTokens` causes rendering issues
7. **Do not use `folderPath: "solution_folder"` for truly external resources** ‚Äî use the actual Orchestrator folder path
8. **Do not copy-paste UUIDs from one resource to another** ‚Äî every resource needs a unique UUID
9. **Do not bump `storageVersion` manually** ‚Äî breaks packager compatibility
10. **Do not call raw Automation.Solutions REST APIs** ‚Äî always use `uip solution` commands

## Completion Output

After completing a task, report:

1. **File paths** ‚Äî which files were created or modified
