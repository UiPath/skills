# state-file

## Purpose
Schema + atomic read/write for `<project>/.dashboard/state.json`. Single source of truth for per-project config that both Build and Deploy share.

> **Path note:** State lives in `<project>/.dashboard/`, NOT `<project>/.uipath-dashboards/`. The OUTER directory `<cwd>/.uipath-dashboards/` is the workspace container that holds all generated dashboard projects (per `intent-capture.md`). To avoid name collision, per-project state uses the shorter `.dashboard/` directory inside each project. This was a deliberate rename — projects now sit at `<cwd>/.uipath-dashboards/<name>/` and their state at `<cwd>/.uipath-dashboards/<name>/.dashboard/state.json`.

## Inputs
- Partial field updates (merged on write).

## Outputs
Atomic on-disk state file.

## Rules
1. **Location:** `<project>/.dashboard/state.json` — NEVER `<project>/.uipath/` (CLI territory) and NEVER `<project>/.uipath-dashboards/` (collides with workspace container name).
2. **Gitignored by default.** Scaffold writes `.gitignore` entry for `.dashboard/` on first Build.
3. **Atomic writes.** Write to `.dashboard/state.json.tmp`, rename to `state.json` on success.
4. **Read is tolerant of missing file** — returns schema defaults (see below).
5. **Schema versioned via `schemaVersion: 1`** for future migrations.

## Schema v1

```json
{
  "schemaVersion": 1,
  "env": "alpha",
  "orgName": "acme",
  "tenantName": "default",
  "folderKey": null,
  "app": {
    "name": "Agent Health Dashboard",
    "routingName": "gov-dashboard-agent-health-dashboard-x7k2",
    "semver": "1.0.0"
  },
  "deployment": {
    "systemName": null,
    "deploymentId": null,
    "deployVersion": null,
    "appUrl": null,
    "deployedAt": null,
    "lastPublishAt": null
  }
}
```

### Field ownership
| Field | Written by | Notes |
|---|---|---|
| `schemaVersion`, `env`, `orgName`, `tenantName` | `intent-capture` (first Build) | derived from `auth-context`; never prompted |
| `folderKey` | **Deploy** by default; Build only if the user's prompt explicitly scopes to a folder | Folder is a Deploy concern — which Orchestrator folder hosts the deployed app. See [../plugins/deploy/impl.md § Step 2](../plugins/deploy/impl.md). |
| `app.name` | `intent-capture` (first Build) | User-friendly Title Case display name (e.g., `Agent Health Dashboard`). |
| `app.routingName` | `intent-capture` (first Build); regenerated by `deploy-cli` only on uniqueness collision | `gov-dashboard-<kebab>-<4-rand>` — the lowercase deploy slug. May be regenerated with a new random suffix if the first deploy fails on routingName uniqueness; persisted on success. |
| `app.semver` | `intent-capture` (first Build), `deploy-cli` (semver bumps) | |
| `deployment.systemName` | `deploy-cli` (first successful deploy) | Server-assigned identifier returned by `uip codedapp deploy`. |
| `deployment.deployVersion` | `deploy-cli` (publish step) | Server-side integer from `uip codedapp publish` `Data.DeployVersion`. |
| `deployment.appUrl` | `deploy-cli` (deploy step) | Live app URL returned by `uip codedapp deploy`. |
| `deployment.deployedAt`, `deployment.lastPublishAt` | `deploy-cli` | Agent wall-clock timestamps in ISO-8601. **Note**: these are agent-side, not server-confirmed. The CLI doesn't currently surface server timestamps; if/when it does, prefer those. |
| `deployment.deploymentId` | **OPTIONAL** post-fresh-deploy — may be `null` until first upgrade | The CLI's `uip codedapp deploy --output json` does not currently return a deployment id. The field stays null after a successful fresh deploy; it gets backfilled the first time an upgrade triggers reconciliation via `assets/scripts/discover-deployment.sh`. Do NOT halt deploy because this is null — it's expected. |
| `deployment.appUrl` | `deploy-cli` (constructed, not from CLI) | The CLI does NOT print a URL on successful deploy. Compute as `https://${orgName}.${envInfix}uipath.host/${routingName}` — that's the only confirmed-working pattern. Other shapes (`/<env>.uipath.com/<org>/<tenant>/apps_/default/<slug>`, `/apps_/run/<systemName>`) look plausible but 404. |

## Magic-string contract: `--description "PINNED"`

Deploy passes `--description "PINNED"` only when the user explicitly opted in (the deploy plan asks if their prompt was silent — see [../plugins/deploy/impl.md § Pin-on-deploy detection](../plugins/deploy/impl.md)). The **Governance Unified Portal** frontend reads the deployment description and pins the dashboard on its home when the value equals exactly `PINNED`. Dashboards never appear on the generic UiPath Apps homepage — pinning only affects the Governance Unified Portal home.

This is an **invisible coupling** between the CLI flag and the Portal's pin-detection logic — there is no schema field that surfaces it. Anyone customizing the deploy command (or the Portal frontend) must know the contract:

- The literal string `PINNED` is the only signal that pins.
- An empty/missing description means "deploy without pinning."
- Other description values are passed through but don't pin.

If the platform later exposes a first-class pin field, retire this magic string. Until then, treat it as load-bearing and document it on any code path that touches the deploy command.

### Atomic write recipe
```bash
DIR="<project>/.dashboard"
mkdir -p "$DIR"
# write new content to .tmp
cat > "$DIR/state.json.tmp" <<EOF
<json content>
EOF
# atomic swap
mv "$DIR/state.json.tmp" "$DIR/state.json"
```

On Windows (Git Bash / PowerShell), `mv` is atomic on the same filesystem. This is the discipline governance uses for audit records.

### Defaults for missing file
Returning schema defaults on read means callers can compose field-updates without pre-existence checks:
```
defaultState = {schemaVersion: 1, env: null, orgName: null, ..., deployment: {all nulls}}
```

### Migration (when v2 schema arrives)
Read `schemaVersion`. If < current, apply migration chain. If > current, halt with "This project was built with a newer skill version; update the skill."

### Error paths
| Condition | Action |
|---|---|
| State file exists but is malformed JSON | Halt: "Invalid state.json; investigate or delete + rebuild." Do NOT silent-overwrite (might lose data). |
| Disk full on write | Halt; .tmp file is left as evidence for debugging. |
| Concurrent write (two Build runs) | Atomic rename means last-writer-wins; no corruption. Document that concurrent Builds are not supported. |
