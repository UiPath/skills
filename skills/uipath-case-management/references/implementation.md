# Implementation Phase: tasks.md ‚Üí caseplan.json

Execute the approved `tasks.md` plan, building `caseplan.json` via either `uip maestro case` CLI commands or direct JSON edits (per-plugin). Validate, then optionally debug or publish.

> **Prerequisite:** The user must have explicitly approved `tasks.md` from the [Planning Phase](planning.md) before starting.
>
> **Input:** `tasks/tasks.md` ‚Äî the complete handoff artifact.

## Strategy selection (per plugin)

Before executing each plugin's T-entries, consult the strategy matrix in [case-editing-operations.md](case-editing-operations.md).

- **`Strategy = JSON`** ‚Üí use the plugin's `impl-json.md` + [case-editing-operations-json.md](case-editing-operations-json.md) for cross-cutting mechanics (ID generation, Pre-flight Checklist, primitive ops).
- **`Strategy = CLI`** ‚Üí use the plugin's `impl-cli.md` + [case-editing-operations-cli.md](case-editing-operations-cli.md) for cross-cutting mechanics.

Mixing strategies within a single skill run is expected during the migration. Both paths conform to the same spec, so output is interchangeable.

> **Per-node-type detail lives in plugins.** This document covers the cross-cutting execution workflow. For how to execute a specific node, consult the matching plugin's `impl-cli.md` or `impl-json.md` per the strategy matrix:
> - Root case ‚Üí `plugins/case/impl-json.md` (migrated) ‚Äî `plugins/case/impl-cli.md` is the fallback
> - Stages ‚Üí `plugins/stages/impl-json.md` (pilot) ‚Äî `plugins/stages/impl-cli.md` is the fallback
> - Edges ‚Üí `plugins/edges/impl-json.md` (JSON strategy) ‚Äî `plugins/edges/impl-cli.md` is the fallback
> - Tasks ‚Üí `plugins/tasks/<type>/impl-cli.md` (or `impl-json.md` for connector-activity and connector-trigger)
> - Triggers ‚Üí `plugins/triggers/<type>/impl-cli.md` (or `impl-json.md` for event triggers)
> - Conditions ‚Üí `plugins/conditions/<scope>/impl-cli.md`
> - SLA ‚Üí `plugins/sla/impl-json.md` (primary) ‚Äî `plugins/sla/impl-cli.md` is the fallback
> - Global variables & arguments ‚Üí `plugins/variables/global-vars/impl-json.md`
> - Task I/O binding ‚Üí `plugins/variables/io-binding/impl-json.md`
> - Logging ‚Üí `plugins/logging/impl-json.md`

## Issue Log ‚Äî Initialize Before Step 6

Before any build step, initialize an empty issue list **in the agent's reasoning** (not as a file, not via subprocess). All plugins append to this shared list during execution. Dump to `tasks/build-issues.md` via the Write tool after Step 12. See [`plugins/logging/impl-json.md`](plugins/logging/impl-json.md) for the entry format, severity levels, and file schema.

```text
# pseudocode ‚Äî kept in the agent's reasoning, not on disk
issues = []  # shared across all steps
```

## Step 6 ‚Äî Create the Case project structure

The case file must live inside a solution + project. Scaffolding commands (solution new ‚Üí case init ‚Üí project add) stay on the CLI path ‚Äî see [`plugins/case/impl-cli.md`](plugins/case/impl-cli.md) ¬ß Prerequisites. After scaffolding, write `caseplan.json` per the migrated JSON recipe in [`plugins/case/impl-json.md`](plugins/case/impl-json.md). Strategy per the matrix in [case-editing-operations.md](case-editing-operations.md) ‚Äî `case` is on the **JSON** strategy; [`plugins/case/impl-cli.md`](plugins/case/impl-cli.md) `cases add` is the fallback.

The JSON recipe writes a pure skeleton: `root` + empty `nodes: []` + empty `edges: []`. **No trigger is emitted at T01.** The primary trigger is added by the triggers plugin at T02 (Step 6.2 / matching trigger plugin's `impl-cli.md`). Its ID will be generated by the triggers plugin ‚Äî there is no `trigger_1` literal to capture from T01.

Under the CLI fallback path, `cases add` still emits the legacy `trigger_1` literal; skip this Step 6.2 coupling only when on the CLI path.

## Step 6.1 ‚Äî Declare global variables and arguments

For each variable/argument T-entry from `tasks.md ¬ß4.2.1`, write entries directly into `caseplan.json` per [`plugins/variables/global-vars/impl-json.md`](plugins/variables/global-vars/impl-json.md). This step populates `root.data.uipath.variables` (inputs, outputs, inputOutputs) and trigger output mappings. Execute these before adding stages ‚Äî downstream tasks and conditions reference variables via `=vars.<id>`.

## Step 7 ‚Äî Add stages

For each stage in `tasks.md ¬ß4.4`, execute per [`plugins/stages/impl-json.md`](plugins/stages/impl-json.md). Strategy per the matrix in [case-editing-operations.md](case-editing-operations.md) ‚Äî currently `stages` is on the **JSON** strategy (pilot); [`plugins/stages/impl-cli.md`](plugins/stages/impl-cli.md) is the fallback. **Capture the generated `StageId` for every stage** into the name ‚Üí ID map (and into `id-map.json` for JSON strategy) ‚Äî downstream edges, tasks, conditions, and SLA all reference it.

`isRequired` from `tasks.md` is planning-only metadata; it is not passed on `stages add`. It is consumed later by case-exit-conditions with `rule-type: required-stages-completed` (Step 10).

## Step 8 ‚Äî Connect stages with edges

For each edge in `tasks.md ¬ß4.5`, execute per [`plugins/edges/impl-json.md`](plugins/edges/impl-json.md). Strategy per the matrix in [case-editing-operations.md](case-editing-operations.md) ‚Äî currently `edges` is on the **JSON** strategy; [`plugins/edges/impl-cli.md`](plugins/edges/impl-cli.md) is the fallback. Edge type is inferred automatically from the source node's `type` in `schema.nodes`.

For multi-trigger cases, add the additional triggers first via the appropriate trigger plugin, then wire their IDs as edge sources.

## Step 9 ‚Äî Add tasks and bind inputs/outputs

For each task entry in `tasks.md ¬ß4.6`, open the matching plugin's implementation doc per the strategy matrix in [case-editing-operations.md](case-editing-operations.md). For all tasks, use `impl-json.md` (direct JSON write). **Capture the `TaskId`** ‚Äî cross-task references and conditions need it.

After adding a task, bind its inputs by editing `caseplan.json` directly per [`plugins/variables/io-binding/impl-json.md`](plugins/variables/io-binding/impl-json.md):

1. Read `caseplan.json`, find the task's `data.inputs[]` by name.
2. For literals/expressions (`input = "<value>"`): write the value string to `input.value`.
3. For cross-task references (`input <- "Stage"."Task".output`): resolve the source output's `var` field from the JSON, then write `=vars.<var>` to the target input's `value`.
4. Write `caseplan.json` back.

For **connector tasks**, pass variable references inline in `--input-values` at creation time (e.g., `"=vars.amount"`). Resolve cross-task `var` IDs from `caseplan.json` before constructing the `--input-values` JSON.

**Binding order.** Process tasks in the order listed in `tasks.md` (already dependency-sorted by `order: after T<n>`). Bind each task's inputs immediately after adding it. If a cross-task reference points to a task not yet added, halt ‚Äî `tasks.md` ordering is wrong; report to the user.

**Pass `--lane <n>` on every task add**, incrementing per task within a stage (starting at 0). Lane is a FE layout coordinate; it does not affect execution. Sequencing and parallelism come from task-entry conditions.

### Step 9.1 ‚Äî Skeleton tasks for unresolved resources

When a task entry's `taskTypeId` (or `type-id` / `connection-id` for connector tasks) is `<UNRESOLVED: ‚Ä¶>`, create a **skeleton task** instead of halting. See [skeleton-tasks.md](skeleton-tasks.md) for the canonical reference.

**Process / agent / rpa / action / api-workflow / case-management (JSON strategy):** follow the Unresolved Fallback section of the matching `plugins/tasks/<type>/planning.md` and write a task with empty `data: {}` (plus `data.taskTitle` / `data.priority` / `data.recipient` for `action`) directly to `caseplan.json` per `plugins/tasks/<type>/impl-json.md`. Omit `data.context`, `data.inputs`, `data.outputs`.

**Connector activity / trigger (CLI strategy):**

```bash
uip maestro case tasks add-connector <file> <stage-id> \
  --type <activity|trigger> \
  --display-name "<name>" \
  --output json
```

**Skip all input binding for skeleton tasks** ‚Äî they have no inputs (empty `data` / no `--task-type-id`). Capture the intended wiring from the fenced `wiring notes` code block in `tasks.md` into the completion report so the user knows what to hook up after registering the resource.

Skeleton tasks integrate with the rest of the graph:
- **Task-entry conditions** use the captured skeleton `TaskId` normally.
- **Stage-exit `selected-tasks-completed`** rules reference skeleton `TaskId`s normally.
- **Cross-task variable bindings** are deferred ‚Äî the user binds them after attaching the real resource.

## Step 11 ‚Äî SLA and escalation

SLA is on the **JSON** strategy per the matrix in [case-editing-operations.md](case-editing-operations.md). Group `tasks.md ¬ß4.8` entries by target (root or stage), then compose and write the full `slaRules[]` array per target in a single mutation per [`plugins/sla/impl-json.md`](plugins/sla/impl-json.md). The JSON path supports per-conditional-rule escalations, ExceptionStage SLA, and multi-recipient single rules ‚Äî all gap-fills over the CLI's capabilities. Fallback to [`plugins/sla/impl-cli.md`](plugins/sla/impl-cli.md) only if the JSON strategy fails empirically.

## Step 12 ‚Äî Validate

```bash
uip maestro case validate <file>
```

On success: `{ Result: "Success", Code: "CaseValidate", Data: { File, Status: "Valid" } }` ‚Äî proceed to Step 13.

On failure: output lists `[error]` and `[warning]` entries with path and message. Fix the reported issues (usually via a targeted re-run of the earlier step) and re-run `validate`.

**Retry policy.** Up to 3 validation retries per session. After the 3rd failure, halt and ask the user with **AskUserQuestion**: show the remaining errors and options ‚Äî `Retry with fix`, `Pause for manual edit`, `Abort`.

## Step 12.1 ‚Äî Dump issue log

Write the issue list to `tasks/build-issues.md` per [`plugins/logging/impl-json.md`](plugins/logging/impl-json.md), grouped by plugin with a summary index. This file is the source of truth for the completion report. Write it even if zero issues were logged (confirms a clean build).

## Step 13 ‚Äî Post-build prompt

Once validation passes, ask the user what to do next.

Use **AskUserQuestion** with options:

- `Run debug session` ‚Äî proceed to Step 14.
- `Publish to Studio Web` ‚Äî proceed to Step 15.
- `Done` ‚Äî exit.
- `Something else` ‚Äî free-form prompt.

After debug or publish completes, return to this prompt so the user can chain the other action (e.g., debug first, then publish). Exit when the user selects `Done`.

For further authoring changes (add a task, tweak a condition, etc.), the user updates `sdd.md` and re-runs the skill from Phase 1 ‚Äî this skill does not offer in-place incremental edits.

## Step 14 ‚Äî Optional: Debug session

> Debug executes the case for real ‚Äî it will send emails, post messages, call APIs, write to databases. Only run debug when the user explicitly asks. Never run it automatically.

```bash
uip maestro case debug "<directory>/<solutionName>/<projectName>" --log-level debug --output json
```

Requires `uip login`. Uploads to Studio Web, runs in Orchestrator, streams results.

## Step 15 ‚Äî Optional: Publish to Studio Web

**Default publish target.** Uploads the case to Studio Web for visualization and editing.

```bash
uip solution upload "<SolutionDir>" --output json
```

Accepts the solution directory (the folder containing the `.uipx`) directly ‚Äî no intermediate bundling step. `upload` pushes to Studio Web ‚Äî share the returned URL with the user.

> **Do NOT run `uip maestro case pack` + `uip solution publish` unless the user explicitly asks for Orchestrator deployment.** That path puts the case directly into Orchestrator, bypassing Studio Web. Default is always Studio Web.
