# Implementation Phase: tasks.md → caseplan.json

Build (or edit) a `caseplan.json` from a `tasks.md` planning document by writing JSON directly. This phase produces a complete, validated case definition ready for `uip case validate` and (with consent) `uip case debug`.

> **Functional parity with `uipath-case-management`.** This guide covers every case construct the CLI-based `uipath-case-management` skill supports — stages, edges, tasks (all 10 types), conditions (4 scopes × all rule types), variable bindings, multi-trigger cases, SLA + escalation, runtime operations. The only difference is *how* the file gets written: this skill writes JSON directly using the [plugins/](plugins/) catalog, whereas `uipath-case-management` calls `uip case stages/tasks/edges/...` commands. Use `uipath-case-management` instead if the user explicitly wants CLI-driven editing.

---

## When to Load This Reference

- A `tasks.md` (planning document) exists or has been provided
- The user wants to **build a new caseplan.json** from a tasks.md
- The user wants to **edit an existing caseplan.json** (rename, add/remove tasks, rewire edges, add conditions, etc.) — see [Common Edits](#common-edits)
- The user wants to **validate, debug, or deploy** a caseplan.json
- The user wants to **manage runtime instances** (list, pause, resume, cancel)

If only a `spec.md` exists and there is no `tasks.md` yet, run [planning.md](planning.md) first to produce the tasks.md.

## Quick Start — New Case from tasks.md

### Step 0 — Resolve the `uip` binary

```bash
UIP=$(command -v uip 2>/dev/null || echo "$(npm root -g 2>/dev/null | sed 's|/node_modules$||')/bin/uip")
$UIP --version
```

Use `$UIP` in place of `uip` if the plain command isn't on PATH.

### Step 1 — Check login

Validation works offline. Registry search, debug, and runtime operations require login.

```bash
uip login status --output json
```

If not logged in and you need cloud features:

```bash
uip login                                          # interactive OAuth
uip login --authority https://alpha.uipath.com     # non-production environments
```

### Step 2 — Scaffold the project (CLI)

The `caseplan.json` must live inside a proper solution/project structure. Use the CLI to create the supporting files (`project.uiproj`, `bindings_v2.json`, `entry-points.json`, `operate.json`, `package-descriptor.json`):

```bash
uip solution new <SolutionName>
cd <SolutionName>
uip case init <ProjectName>
uip solution project add <ProjectName> <SolutionName>.uipx
```

> **Naming convention:** Reuse the same name for solution and project unless the user specifies otherwise.

See [plugins/setup/case-skeleton](plugins/setup/case-skeleton/impl.md) for the full scaffold layout.

### Step 3 — Pull the registry (only if tasks.md needs validation)

If tasks.md was generated by [planning.md](planning.md), all `taskTypeId` values are already resolved. You only need a fresh registry pull when:

- The user wants to validate that resolved IDs still exist
- A task references a connector or app whose `resourceKey`/`folderPath` is missing from tasks.md

```bash
uip case registry pull
uip case registry search "<task display name>" --output json    # capture resourceKey + folderPath
```

### Step 4 — Read the tasks.md

Parse the `tasks.md` end-to-end before writing JSON. The planning format groups work by concern and by numbered task (`T01`, `T02`, …):

| Planning group | What to build |
|---|---|
| `Create case file "<name>"` | Root node — name, caseIdentifier, caseIdentifierType, description |
| `Configure <type> trigger "<name>"` | Trigger node (manual is implicit; timer / connector explicit) |
| `Create stage "<name>"` / `Create exception stage "<name>"` | Stage / ExceptionStage node — label, isRequired, description |
| `Add edge "<A>" → "<B>"` | TriggerEdge or Edge depending on source type |
| `Add <type> task "<name>" to "<stage>"` | Task in that stage — declare bindings, write task |
| `Add stage entry condition for "<stage>"` | `stage.data.entryConditions[]` |
| `Add stage exit condition for "<stage>"` | `stage.data.exitConditions[]` |
| `Add task entry condition for "<task>" in "<stage>"` | `task.entryConditions[]` |
| `Add case exit condition` | `root.caseExitConditions[]` |
| `Set default SLA for <scope> to N <unit>` | `slaRules[].count` / `unit` on root or stage |
| `Add conditional SLA rule for <scope>` | Additional `slaRules[]` entry with `expression` |
| `Add escalation rule for <scope>` | `slaRules[].escalationRule[]` entry |

Capture every name → ID assignment as you build (StageId, TaskId, ConditionId, …) so later sections can reference them.

### Step 5 — Build caseplan.json in one pass

Load [plugins/setup/case-skeleton](plugins/setup/case-skeleton/impl.md) first for the root JSON skeleton, then load other plugins on demand.

**Build order (do not interleave):**

1. **Root** — name, identifier, identifierType, description, empty `bindings`/`variables`/`slaRules`/`caseExitConditions`. See [plugins/setup/case-skeleton](plugins/setup/case-skeleton/impl.md).
2. **Trigger nodes** — manual is created by default. Add timer or connector triggers if tasks.md declares them. See [plugins/triggers/manual](plugins/triggers/manual/impl.md), [plugins/triggers/timer](plugins/triggers/timer/planning.md), [plugins/triggers/connector-trigger](plugins/triggers/connector-trigger/planning.md).
3. **Stage nodes** (and ExceptionStages) — empty tasks/conditions arrays. See [plugins/setup/stage](plugins/setup/stage/impl.md), [plugins/stage-types/exception-stage](plugins/stage-types/exception-stage/planning.md).
4. **Edges** — Trigger → first stage (TriggerEdge), then Stage → Stage (Edge). Vertical branches for ExceptionStages. See [plugins/setup/edge](plugins/setup/edge/impl.md).
5. **Tasks per stage** — for each task in tasks.md, look up the right plugin, declare its bindings at the root, and write the task into the stage's `tasks[]`. See [Task type plugins](#task-type-plugins) below.
6. **Conditions** — fill `entryConditions` / `exitConditions` on each stage and task; fill `caseExitConditions` on root. See [Condition plugins](#condition-plugins) below.
7. **Variable bindings** — wire literal values and cross-task data flow into task inputs. See [Variable Binding](#variable-binding) below.
8. **SLA + escalation** — set `slaRules` on root and per-stage; add escalation entries. See [plugins/sla/setup](plugins/sla/setup/planning.md).

> **For connector tasks** (`execute-connector-activity`, `wait-for-connector`, connector triggers, `external-agent`): write a placeholder task and proceed. These cannot be fully hand-written — Phase 6 enriches them via CLI.

### Step 6 — Enrich connector tasks (CLI)

Connector tasks and event triggers require additional metadata that only the CLI can produce. Run these *after* the rest of the JSON is in place:

```bash
uip case tasks add-connector <file> <stage-id> --type activity \
  --type-id <typeId> --connection-id <connectionId> --output json

uip case tasks add-connector <file> <stage-id> --type trigger \
  --type-id <typeId> --connection-id <connectionId> --output json

uip case triggers add-event <file> \
  --type-id <typeId> --connection-id <connectionId> --output json
```

`<typeId>` is the `uiPathActivityTypeId` recorded by [planning.md](planning.md). `<connectionId>` comes from `uip case registry get-connection`. See [plugins/tasks/execute-connector-activity](plugins/tasks/execute-connector-activity/planning.md), [plugins/tasks/wait-for-connector](plugins/tasks/wait-for-connector/planning.md), [plugins/tasks/external-agent](plugins/tasks/external-agent/planning.md).

### Step 7 — Validate

```bash
uip case validate <path/to/caseplan.json> --output json
```

On `Result: Success` and `Status: Valid` → proceed. On failure, the output lists each `[error]` / `[warning]` with a JSON path. Fix each and re-run. **Validation loop max 5 attempts** — if errors persist, surface the issue to the user instead of guessing.

### Step 8 — Ask about debug

Once validation passes, tell the user and ask before running debug:

> "Case file built and validated. Do you want to debug it? Debug uploads to Studio Web and runs the case in Orchestrator with real side effects — emails, API calls, database writes."

Use `AskUserQuestion` with options "Yes" / "No". Do **not** run debug without explicit consent (Critical Rule #10 in SKILL.md).

### Step 9 — Debug (only on consent)

```bash
uip case debug --project-dir <projectDir> --output json
```

The argument is the project directory containing `project.uiproj`. Requires `uip login`.

## Common Edits — Existing caseplan.json

Use these recipes for targeted changes without rebuilding the whole file. Run `uip case validate` once after each batch of edits.

### Rename a stage or task

Edit the `data.label` (stage) or `displayName` (task) field. **Do not change the `id`** — every reference (edges, conditions, elementId) depends on it.

### Add a task to an existing stage

1. Generate a new task ID (`t` + 8 chars) and `elementId` (`<stageId>-<taskId>`).
2. Look up the resource via `uip case registry search "<name>" --output json` to get `resourceKey`, `name`, `folderPath`.
3. Declare the binding pair at `root.data.uipath.bindings`. See [plugins/variables/bindings](plugins/variables/bindings/impl.md).
4. Build the task body using the task-type plugin's `impl.md`.
5. Append the task as its own lane entry in `stage.data.tasks[]` (one task per lane — never group).
6. If outputs are declared, add the global variable to `root.data.uipath.variables.inputOutputs`.

### Remove a task from a stage

1. Delete the task entry from `stage.data.tasks[]`.
2. Delete its bindings from `root.data.uipath.bindings` (only if no other task uses the same `id`).
3. Delete its global outputs from `root.data.uipath.variables.inputOutputs`.
4. Scan all conditions for `selectedTasksIds` references and remove the task ID from those arrays.
5. If the array becomes empty, the condition is no longer satisfiable — replace with `required-tasks-completed` or remove the condition.

### Insert a stage between two existing stages

1. Add the new stage node (see [plugins/setup/stage](plugins/setup/stage/impl.md)). Set `position.x` halfway between the neighboring stages.
2. Find the existing edge from `<source>` → `<originalTarget>` and update its `target` + `targetHandle` to the new stage.
3. Add a new edge from the new stage to `<originalTarget>` with `data: {}` and `type: "case-management:Edge"`.
4. If conditions reference the old transition (e.g., `selected-stage-completed` with `selectedStageId: <originalTarget>`), update the relevant `selectedStageId` to the new stage.

### Rewire an edge

Edit `source` / `target` / `sourceHandle` / `targetHandle` directly. The handle format is `<nodeId>____source____<direction>` (4 underscores each side). See [plugins/setup/edge](plugins/setup/edge/impl.md).

### Add a stage entry/exit condition to an existing stage

Append to `stage.data.entryConditions[]` or `stage.data.exitConditions[]` using the appropriate plugin: [plugins/conditions/stage-entry](plugins/conditions/stage-entry/planning.md), [plugins/conditions/stage-exit](plugins/conditions/stage-exit/planning.md). Generate fresh `Condition_` and `Rule_` IDs.

### Add a task entry condition

Append to `task.entryConditions[]`. Default rule for parallel-on-stage-entry is `current-stage-entered`. For sequential ordering inside a stage, use `selected-tasks-completed` with the upstream task's ID. See [plugins/conditions/task-entry](plugins/conditions/task-entry/planning.md).

### Add or change a case exit condition

Append to `root.caseExitConditions[]`. Standard pattern is `required-stages-completed` with `marksCaseComplete: true`. **Never use `required-tasks-completed` here** — that rule is stage-scoped only. See [plugins/conditions/case-exit](plugins/conditions/case-exit/impl.md).

### Add an SLA or escalation to root or a stage

Add to `root.data.slaRules[]` or `stage.data.slaRules[]`. Conditional rules go *before* the default `=js:true` rule (evaluation is ordered). Each rule has its own `escalationRule[]` array. See [plugins/sla/setup](plugins/sla/setup/planning.md).

### Add a workflow-level variable

Add to `root.data.uipath.variables.inputs` (input from caller), `outputs` (returned to caller), or `inputOutputs` (case-internal). Reference with `=vars.<id>` from task inputs or condition expressions. See [plugins/variables/global-vars](plugins/variables/global-vars/planning.md).

### Add a multi-trigger entry point

A default manual trigger is created automatically. To add another entry point (typically a timer or connector trigger):

1. Add a new trigger node with a unique ID. Stack vertically — first trigger at `y: 0`, additional triggers at `y: 150`, `y: 300`, etc.
2. Add a `TriggerEdge` from the new trigger to the stage that should activate first when it fires.
3. For timer triggers, see [plugins/triggers/timer](plugins/triggers/timer/planning.md). For connector triggers, see [plugins/triggers/connector-trigger](plugins/triggers/connector-trigger/planning.md) (requires Step 6 enrichment).

## Variable Binding

Every task input or expression in the case file is one of the forms below. All prefixes are **defined in `VariableConstants.ts`** in the FE — anything outside this set will not be resolved by the runtime.

| Form | Prefix constant | Purpose | Where it's typically used |
|---|---|---|---|
| `=vars.<varId>` | `VARIABLES_PATH_PREFIX` | Read a global variable (`root.data.uipath.variables.*`) | task `input.value` (literal read), legacy `conditionExpression` |
| `=bindings.<bindingId>` | `BINDINGS_PATH_PREFIX` | Reference a root binding (process name, folder path, app name, connection ID) | task `data.name`, `data.folderPath` — agents rarely hand-write this elsewhere |
| `=js:<expression>` | `JAVASCRIPT_PATH_PREFIX` | Evaluate as JavaScript. Use `vars.x` inside (no `$`). | `conditionExpression` on rules, SLA `expression`, any computed input |
| `=metadata.<field>` | (runtime-resolved, no constant) | Read case-instance metadata (set by the platform at case creation, NOT by task outputs) | task inputs, conditions |
| `=datafabric.<entity.field>` | `DATA_FABRIC_PREFIX` | Read or write a Data Fabric entity field | task `input.value`, output `target` |
| `=orchestrator.JobAttachments` | `JOB_ATTACHMENT_PREFIX` | Orchestrator job-attachment storage handle for file outputs | task output `target` for file/binary outputs |
| `=response.<field>` | (output-context, no constant) | Reference a field of the task's own response object | task output `source` only — when the task returns a structured response (API workflow, agent) |
| `=string.Format("<fmt>", arg, ...)` | (no constant — C#-style template) | String template using `{0}`, `{1}` placeholders. **Not JavaScript.** | `taskTitle`, `caseAppConfig.caseSummary` |
| `<plain text>` (no prefix) | — | Literal value passed verbatim | task `input.value` for static strings |

### Available `=metadata.*` fields

Confirmed in FE source / test fixtures:

| Field | Type | Meaning |
|---|---|---|
| `InstanceId` | string | Case instance UUID — unique per running case |
| `FolderKey` | string | Orchestrator folder UUID containing the case |
| `ProcessKey` | string | Case definition's process key |
| `RetryCount` | number | Retry counter for the current execution |
| `caseType` | string | Case type identifier |
| `priority` | string | Case priority (when set externally) |
| `status` | string | Current case status |
| `config` / `configValue` | any | Configuration values passed when starting the case |

### Combining forms inside `=js:`

Inside a `=js:` expression you can chain `vars.x.field` for nested object access, use comparison/logical operators, and call standard JavaScript functions:

```
=js:vars.response.amount > 50000
=js:(vars.decision == "approve" && vars.score > 0.8)
=js:vars.items.length === 0
=js:vars.response.tags.includes("urgent")
```

You **cannot** mix expression forms — `=js:bindings.x` and `=metadata.<js-expr>` are not valid. To use a metadata value in JS, the runtime exposes it directly: `=js:metadata.InstanceId === vars.expectedInstance`.

### Nested object access

Task outputs may be **objects, not scalars**. When a task declares an output of `type: "jsonSchema"` (or returns an object from an agent / API workflow), downstream consumers reference fields with dot notation:

```json
// Source task output declaration (object)
{ "name": "response", "type": "jsonSchema", "var": "response", "id": "response", ... }

// Downstream task input reading a field of that object
{ "name": "customerName", "value": "=vars.response.customerName", "type": "string", ... }

// Or in caseAppConfig
"caseSummary": "=string.Format(\"{0} - {1}\", vars.response.customerName, vars.response.policyId)"
```

Object outputs do **not** need a separate `inputOutputs` entry per field — declaring the parent variable once is sufficient; field access is resolved at runtime.

### `=` vs `=js:` quick rule

- **Just reading a variable?** Use `=vars.x` (no `js:`)
- **Evaluating a condition / expression?** Use `=js:vars.x === 'foo'`
- **Inside a `conditionExpression` field on a rule?** Use `=js:` (always JavaScript)
- **Inside a string template (taskTitle, caseSummary)?** Use `=string.Format(...)`

### Bind a literal value to a task input

Set `input.value` to the raw string (for plain text), or to an `=js:...` expression for typed values:

```json
{
  "name": "threshold",
  "displayName": "threshold",
  "value": "=js:50000",
  "type": "number",
  "id": "threshold",
  "elementId": "<stageId>-<taskId>"
}
```

### Wire a source-task output to a downstream task input

The source task must declare the output in its `data.outputs[]` AND a global variable must exist in `root.data.uipath.variables.inputOutputs`.

1. **Source task — declare output:**
   ```json
   {
     "name": "claimAmount", "displayName": "claimAmount", "value": "claimAmount", "type": "number",
     "source": "=claimAmount", "var": "claimAmount", "id": "claimAmount", "target": "=claimAmount",
     "elementId": "<sourceStageId>-<sourceTaskId>"
   }
   ```

2. **Root — declare global variable:**
   ```json
   { "id": "claimAmount", "name": "claimAmount", "displayName": "claimAmount", "type": "number" }
   ```

3. **Target task — reference the variable in its input:**
   ```json
   {
     "name": "amount", "displayName": "amount", "value": "=vars.claimAmount", "type": "number",
     "id": "amount", "elementId": "<targetStageId>-<targetTaskId>"
   }
   ```

See [plugins/variables/bindings](plugins/variables/bindings/impl.md) and [plugins/variables/global-vars](plugins/variables/global-vars/planning.md).

## Condition Plugins

Conditions live inside the element they govern. All four scopes share the `rules: [[{ rule, id, ... }]]` shape (outer array = OR groups, inner array = AND group).

| Scope | Lives in | Common rules | Plugin |
|---|---|---|---|
| Stage entry | `stage.data.entryConditions[]` | `case-entered`, `selected-stage-completed`, `selected-stage-exited`, `current-stage-entered`, `adhoc`, `wait-for-connector` | [plugins/conditions/stage-entry](plugins/conditions/stage-entry/planning.md) |
| Stage exit | `stage.data.exitConditions[]` | `required-tasks-completed`, `selected-tasks-completed`, `wait-for-connector` | [plugins/conditions/stage-exit](plugins/conditions/stage-exit/planning.md) |
| Task entry | `task.entryConditions[]` | `current-stage-entered`, `selected-tasks-completed`, `adhoc`, `wait-for-connector` | [plugins/conditions/task-entry](plugins/conditions/task-entry/planning.md) |
| Case exit | `root.caseExitConditions[]` | `required-stages-completed`, `selected-stage-completed`, `selected-stage-exited`, `wait-for-connector`, `adhoc` | [plugins/conditions/case-exit](plugins/conditions/case-exit/impl.md) |

> **`wait-for-connector`** has complex metadata that should be enriched via CLI rather than hand-written. See [plugins/conditions/wait-for-connector](plugins/conditions/wait-for-connector/impl.md) for full schema and enrichment steps.

**Stage exit `type` field** (full set, per FE `CaseManagementNodeExitRuleType`):

| `type` | Companion fields | Meaning |
|---|---|---|
| `exit-only` | — | Default. Stage exits and case continues along the edge. |
| `wait-for-user` | — | Stage waits for a human to pick the next stage. |
| `return-to-origin` | — | Returns to the calling stage (ExceptionStages, re-entry loops). |
| `terminal` | — | Stage completion **ends the entire case** directly. |
| `send-to-stage` | `exitToStageId` (required) | Explicitly routes to a chosen stage, overriding edge wiring. |
| `rework-stage-and-return` | `exitToStageId` (required) | Branches to another stage for rework, then returns to this one. |

**Conditional filtering:** Any rule can carry a `conditionExpression` that gates activation. Canonical form is **`=js:` prefix + `vars.x`** access (matches FE `addJavascriptPrefix` output). Legacy `=vars.x ...` (no `js:`) is still parsed by the runtime but new code should emit `=js:`:

```json
{ "rule": "selected-stage-completed", "id": "Rule_xxxxxx", "selectedStageId": "Stage_xxxxxx", "conditionExpression": "=js:vars.priority === 'high'" }
```

> The variable accessor is `vars.x` (no `$`). The `$vars.x` syntax does not exist in the runtime.

## SLA and Escalation

`slaRules` is an array on `root.data` and on each `stage.data`. Rules evaluate top-down — first matching `expression` wins. The fallback is always `expression: "=js:true"`.

```json
"slaRules": [
  {
    "expression": "=js:vars.priority === 'high'",
    "count": 1, "unit": "d",
    "escalationRule": [
      { "id": "esc_xxxxxx", "displayName": "High-Priority At-Risk 80%",
        "action": { "type": "notification", "recipients": [{ "scope": "User", "target": "<uuid>", "value": "urgent@co.com" }] },
        "triggerInfo": { "type": "at-risk", "atRiskPercentage": 80 } }
    ]
  },
  { "expression": "=js:true", "count": 5, "unit": "d", "escalationRule": [] }
]
```

| Trigger type | Required field |
|---|---|
| `at-risk` | `atRiskPercentage` (1–99) |
| `sla-breached` | (none) |

| Recipient scope | `target` value |
|---|---|
| `User` | User UUID |
| `UserGroup` | Group UUID |

| Time unit | Meaning |
|---|---|
| `h` | Hours |
| `d` | Days |
| `w` | Weeks |
| `m` | Months |

See [plugins/sla/setup](plugins/sla/setup/planning.md) for full patterns including stage-level SLA, no-escalation minimal rules, and conditional escalation.

## Condition Expression Translation

When tasks.md contains human-readable conditions, translate them to `=js:` expressions:

### Translation Rules

| Human-readable (tasks.md) | JSON `conditionExpression` |
|---------------------------|---------------------------|
| `Priority = Urgent` | `=js:vars.priority === 'Urgent'` |
| `decision is "Approve"` | `=js:vars.decision === 'Approve'` |
| `amount > 50000` | `=js:vars.amount > 50000` |
| `status != "Pending"` | `=js:vars.status !== 'Pending'` |
| `counter > 0` | `=js:(vars.counter != null && vars.counter > 0)` |
| `counter is 0 or null` | `=js:(vars.counter == null \|\| vars.counter == 0)` |
| `tags includes "urgent"` | `=js:vars.tags.includes('urgent')` |
| `items is empty` | `=js:vars.items.length === 0` |
| `response.customerName exists` | `=js:vars.response.customerName != null` |

### Key Syntax Rules

1. **Always prefix with `=js:`** — the runtime requires this for evaluation
2. **Use `vars.` accessor** — not `$vars.` or bare variable names
3. **String comparisons use `===`** — JavaScript strict equality
4. **Null checks use `== null`** — catches both `null` and `undefined`
5. **Logical operators:** `&&` (and), `||` (or), `!` (not)
6. **Nested object access:** `vars.response.field` (dot notation)
7. **Parenthesize compound expressions** — `=js:(vars.a && vars.b) || vars.c`

### Re-entry Counter Expressions

For stages with re-entry behavior, use the counter pattern:

| Condition | Expression |
|-----------|------------|
| First run only | `=js:(vars.finishedRunCountIntake == null \|\| vars.finishedRunCountIntake == 0)` |
| Re-entry only | `=js:(vars.finishedRunCountIntake != null && vars.finishedRunCountIntake > 0)` |

### Combined Conditions

When multiple conditions must be met:

```json
"conditionExpression": "=js:(vars.decision == 'Reject') && (vars.finishedRunCountIntake == null || vars.finishedRunCountIntake == 0)"
```

## Runtime Operations (CLI)

After the case is deployed, manage live instances via CLI. All require `uip login` and `--output json` for parsing.

```bash
# Inspect deployed case definitions
uip case process list --output json
uip case process get <processKey> --output json
uip case incidents list --output json

# Manage running case instances
uip case instance list --process-key <key> --output json
uip case instance get <instanceId> --output json
uip case instance pause <instanceId> --output json
uip case instance resume <instanceId> --output json
uip case instance cancel <instanceId> --output json

# Inspect or update instance variables
uip case instance var list <instanceId> --output json
uip case instance var set <instanceId> <varId> --value <value> --output json
```

See [commands-reference.md](commands-reference.md) for the full flag reference.

## Anti-Patterns

- **Do not call `uip case stages/tasks/edges/sla/conditions/var` add commands** — write JSON directly. The CLI versions exist in `uipath-case-management`; they are not used here.
- **Do not group multiple tasks in the same lane** — one task per lane (`tasks: [[t0],[t1],[t2]]`). Parallelism is controlled by entry conditions, never lane grouping. The 2D `[[t0,t1]]` form is deprecated.
- **Do not hand-write connector task schemas** — use `tasks add-connector` (Step 6).
- **Do not reuse IDs** — every element needs a unique ID. ID format rules are in [SKILL.md → ID Generation](../SKILL.md#id-generation).
- **Do not omit `elementId` on tasks or task inputs/outputs** — format is `<stageId>-<taskId>`. Validation rejects missing or mismatched values.
- **Do not use `required-tasks-completed` on case exit conditions** — that rule is stage-scoped. Use `required-stages-completed`.
- **Do not omit `"data": {}` on edges** — required on both `case-management:TriggerEdge` and `case-management:Edge`.
- **Do not skip validation before debug** — always run `uip case validate` first; fix all errors before debug.
- **Do not run debug without explicit consent** — debug executes the case for real (emails, API calls, DB writes).
- **Do not edit `*.bpmn` files** — auto-generated from caseplan.json; will be overwritten.
- **Do not validate after every individual JSON edit** — intermediate states may legitimately be invalid. Run validate once after a complete batch of edits.
