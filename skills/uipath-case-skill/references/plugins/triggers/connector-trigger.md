# Connector Trigger

A connector trigger starts a case automatically when an external event fires (e.g., new Salesforce record, ServiceNow ticket update, Jira webhook).

## When to Use

- Case starts when an external system event occurs ā€” **use this**
- Case starts on a schedule ā€” [timer trigger](timer.md)
- Case starts manually ā€” [manual trigger](manual.md)

If no connector trigger exists for the needed event, fall back to timer trigger + polling process.

## Prerequisites

- `uip login` required ā€” connector trigger types only appear after authentication
- A healthy IS connection must exist for the connector (user must create one if missing)
- The connector must support event triggers (not all do)

## Event Modes

Determined by the connector (not configurable):

| Mode | Behaviour |
|---|---|
| `polling` | Runtime polls the service on an interval ā€” slight delay |
| `webhooks` | Near-real-time via registered webhook |

Note the event mode in the plan for the user's awareness.

## Planning Annotation

Annotate in the plan as: `trigger: <service> ā€” <event>` (e.g., "trigger: Jira ā€” issue created")

## Implementation

### Why CLI Enrichment Is Required

The connector trigger node is **not fully hand-writable** because `data.uipath.context[].metadata.body.activityPropertyConfiguration.configuration` is a serialized snapshot of the IS designer's internal state ā€” produced by the connector's introspection API, not user-authored JSON.

That `=jsonString:{...}` value carries `instanceParameters`, designer caches (`cachedLookupValues`, `cachedBrowserItems`, `cachedPropertyMappings`), output/event JSON schemas, field metadata (~25 fields x N fields), and filter UI state. Hand-writing this is brittle. **Use the CLI.**

What **is** hand-writable:

| Part | Hand-writable? | Notes |
|---|---|---|
| Trigger node `id` / `position` / `data.label` | Yes | Standard node skeleton |
| `data.uipath.serviceType` | Yes | Always `"Intsvc.EventTrigger"` |
| Root binding pair (`Connection` + `folderKey`) | Yes | See [bindings](../variables/bindings.md) |
| TriggerEdge from trigger to first stage | Yes | See [edge](../skeleton/edge.md) |
| `data.uipath.context[]` metadata block | No | Generated by `add-event` |
| `data.uipath.inputs[].body` | Partial | Outer shape is hand-writable; `parameters` depends on connector |
| `data.uipath.outputs[]._jsonSchema` | No | From connector's describe response |

### Step 1 ā€” Find the Connector and Connection

```bash
uip case registry pull
uip case registry search "<service name> trigger" --output json
uip case registry get-connection --key <connectorKey> --output json
```

Capture:
- `typeId` ā€” the `uiPathActivityTypeId` from the registry search result
- `connectionId` ā€” UUID of the enabled connection

If no connection exists, tell the user ā€” they must create one in the IS portal.

### Step 2 ā€” Preview the Trigger Schema (Optional but Recommended)

```bash
uip case tasks describe \
  --type connector-trigger \
  --id <typeId> \
  --connection-id <connectionId> \
  --output json
```

Shows `inputs` (event parameters with `requiredFields`) and `outputs` (event payload fields). Use this to prepare `--event-params` in Step 3.

### Step 3 ā€” Add the Enriched Event Trigger

```bash
uip case triggers add-event <file> \
  --type-id <typeId> \
  --connection-id <connectionId> \
  --output json
```

With event parameters:
```bash
uip case triggers add-event <file> \
  --type-id <typeId> \
  --connection-id <connectionId> \
  --event-params '{"project": "PROJ", "issuetype": "Bug"}' \
  --output json
```

With filter expression:
```bash
uip case triggers add-event <file> \
  --type-id <typeId> \
  --connection-id <connectionId> \
  --filter "((fields.status<`In Progress`>))" \
  --output json
```

> `add-event` **adds** a new trigger node. If a manual trigger already exists and should be replaced, remove it from `nodes` and its edge from `edges` before running this command.

Capture the trigger node ID from the output.

### Step 4 ā€” Wire the Trigger to the First Stage

`add-event` adds the trigger node but not its edge. Add a `TriggerEdge` manually ā€” see [edge reference](../skeleton/edge.md).

### What the CLI Writes

After enrichment, the trigger node's `data.uipath` is populated:

```json
{
  "id": "trigger_<6chars>",
  "type": "case-management:Trigger",
  "position": { "x": 0, "y": 0 },
  "data": {
    "label": "Trigger 1",
    "uipath": {
      "serviceType": "Intsvc.EventTrigger",
      "context": [
        { "name": "connectorKey", "value": "uipath-atlassian-jira", "type": "string" },
        { "name": "connection", "value": "=bindings.<connectionBindingId>", "type": "string" },
        { "name": "resourceKey", "value": "<connectionId>", "type": "string" },
        { "name": "folderKey", "value": "=bindings.<folderKeyBindingId>", "type": "string" },
        { "name": "operation", "value": "ISSUE_CREATED", "type": "string" },
        { "name": "objectName", "value": "Issue", "type": "string" },
        { "name": "method", "value": "", "type": "string" },
        { "name": "path", "value": "", "type": "string" },
        {
          "name": "metadata",
          "type": "json",
          "body": {
            "activityPropertyConfiguration": {
              "objectName": "Issue",
              "eventType": "ISSUE_CREATED",
              "eventMode": "polling",
              "configuration": "=jsonString:{...}",
              "uiPathActivityTypeId": "<typeId>"
            }
          }
        }
      ],
      "inputs": [
        {
          "name": "body",
          "type": "json",
          "body": {
            "filters": { "expression": "" },
            "parameters": {}
          }
        }
      ],
      "outputs": [
        {
          "name": "response",
          "displayName": "Issue Created",
          "type": "jsonSchema",
          "source": "=response",
          "_jsonSchema": { "..." : "..." }
        },
        { "name": "Error", "type": "json", "source": "=error" }
      ],
      "bindings": []
    }
  }
}
```

Root bindings are also appended:
```json
{ "id": "b<8chars>", "name": "connection", "resource": "connection", "resourceKey": "<connectionId>" },
{ "id": "b<8chars>", "name": "folderKey", "resource": "folderKey", "resourceKey": "<folderKey>" }
```

Trigger outputs are added to `root.data.uipath.variables.inputOutputs` for BPMN conversion.

## Common Errors

| Error | Cause | Fix |
|---|---|---|
| Trigger type not found | Not authenticated or registry stale | `uip login` then `uip case registry pull` |
| `connection not found` | Connection ID incorrect or expired | Re-run `registry get-connection` |
| No connection exists | User hasn't created one | User must create connection in IS portal |
| `filter` syntax error | Wrong format | Use JMESPath: `((fields.<fieldName><\`value\`>))` |
| Trigger fires on all events | Filter missing or incorrect | Check `describe` output for filter fields |
| Edge missing after add-event | `add-event` doesn't create edges | Add `TriggerEdge` manually (Step 4) |
