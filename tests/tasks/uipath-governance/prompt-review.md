# `tests/tasks/uipath-governance/` — prompt review

Existing test prompts vs. natural-user rewrites. Methodology in [hitl-prompts-review.html](../../hitl-prompts-review.html) and [CLAUDE.md](../../CLAUDE.md).

## Scope of this folder

The `uipath-governance` skill is the uber-skill for UiPath governance authoring, exposing two CLI surfaces: AOps **product policies** (`uip gov aops-policy` — what Studio / StudioX / Assistant / Robot / AI Trust Layer / Agent Builder can do) and Access **ToolUsePolicy** (`uip gov access-policy` — which Actor Processes / identities can invoke which child Resource). The skill's first job is to *classify* an ambiguous request into one of these two branches before authoring. This folder has 11 yaml tests: 4 under `access-policy/` (one e2e lifecycle, three smoke) and 7 under `aops-policy/` (one e2e lifecycle, six smoke). Smoke tests run in an unauthenticated sandbox and exist almost exclusively to verify the agent reaches for the right CLI namespace / subcommand / flags; the two e2e tests run end-to-end CRUD against a staging tenant.

## Insider markers seen in this folder

- **Eval-harness contract paragraphs** verbatim in every smoke prompt: "Save the command output to `./output.txt`", "use `> output.txt 2>&1` for the first command and `>> output.txt 2>&1` for subsequent ones", "Run each command once...and move on. Do NOT retry or attempt to login", "Do NOT prompt the user for confirmation — this is an automated test".
- **CLI-flag literacy** baked into user voice: `--output json` ("Use --output json on every uip gov aops-policy command"), `--filter`, `--order-by`, `--output-filter`, `--input`, `--source`, `--user`, `--email`, `--group`, `--product-name`, `--output-dir`.
- **CLI namespace named in the prompt**: "every uip gov access-policy command", "every uip gov aops-policy command".
- **Synthetic test-fixture identifiers**: zero-padded UUIDs (`00000000-0000-0000-0000-000000000001`, `...00a1`, `...00b2`, `11111111-1111-...`, `22222222-2222-...`), `alice@example.com`, `build-bot`, `Alice Example`, `Ops Team`, `cloud` / `aad` source literals.
- **Schema vocabulary leaked into the goal**: "full PolicyDefinition (selectors, rules, enforcement)", "three UUIDs for the `actorRule.values[]` array", "minimal Simulated Access ToolUsePolicy", "one resource selector for `Agent` matching all resources / one actor-process rule for `Agent` matching all callers", "policy GUID", "license type / product / tenant tuple".
- **Skill-internal callouts in the user voice**: "The uipath-governance skill knows the right CLI surface ... Trust it." (both e2e prompts), "full-replace semantics of update", "what to re-pass to preserve fields".
- **Test-purpose verbs**: "If `deployment user list` fails with auth error, still call `deployment user get <PLACEHOLDER_ID>` once with any placeholder user identifier so the list→get chain is exercised" (the user telling the agent to exercise a code path).
- **Numbered step lists with structured field blocks** (deployment_configure_smoke: `USER_ID` / `Display name` / `Email` / `Source` / `Assignment`) — a real admin would describe these as a sentence, not a config block.

## Verdict summary

| Verdict | Count |
|---|---|
| Insider — fixable | 4 |
| Insider — legitimate (CLI/refusal/antipattern coverage) | 0 |
| Mixed | 7 |
| Natural | 0 |

**Folder-level finding:** every smoke test in this folder is, by construction, a CLI-flag-coverage test — checking that the agent reaches for the right subcommand and flag set without an authenticated tenant. The *eval-harness contract* (write to `./output.txt`, don't retry on auth errors, don't prompt) is legitimately needed and should stay. What can and should be humanized is the **business-intent framing** wrapped around that contract: "show me every active access policy, sorted by name" instead of "ordered by policy name ascending. Use --order-by". Most smoke tests are therefore **Mixed** — the eval-contract block stays as a small footer, the user-intent body becomes natural. The two e2e tests are mostly natural in shape but leak skill-internal vocabulary and explicit "trust the skill" stage directions; those are **fixable**.

## Notes for the PR description

- **No tests in this folder are pure flag-coverage / refusal / antipattern in a way that *requires* insider prompts** — every smoke test pairs a CLI-coverage signal with a user-intent framing that could be expressed naturally. The eval-harness contract footer (write to `./output.txt`, expect auth errors, don't retry) is legitimate test-fixture machinery and should stay, but it can be moved into a parenthetical note instead of being interleaved with the user's request.
- **Recurring leakage pattern: the prompt names the flag the test is actually grading.** `list_policies_smoke` asks for sort + filter while requiring `--filter` / `--order-by`; `identity_lookup_smoke` literally tells the agent which retired paths *not* to use; `template_bootstrap_smoke` uses the verb "Bootstrap" and the exact `--output-dir` path. The agent's *judgment* is supposed to be what's measured — if the prompt names the flag, the test is measuring obedience.
- **Both e2e tests end with "Trust [the skill]."** This is an unusual stage direction to put in a user prompt and a strong tell that the prompt was written by someone who can see both sides of the skill contract. Real customers don't say "trust the skill" — they describe what they want and assume the agent will figure it out.
- **Schema vocabulary worth flagging to the skill author:** "Simulated Access ToolUsePolicy", "minimal `PolicyDefinition`", "selectors, rules, enforcement", "`actorRule.values[]` array", "policy GUID", "license type / product / tenant tuple". These are valid internal names from `access-policy/`, `aops-policy/` reference guides — but in a customer voice they should be "simulation-mode policy", "what this policy covers", "the three identities the policy applies to", "policy id", "this user on this product".
- **`deployment_discovery_smoke` has a clear meta-leak**: "still call `deployment user get <PLACEHOLDER_ID>` once with any placeholder user identifier so the list→get chain is exercised." That is the test author talking to the agent through the user's mouth. Worth rewriting in the user voice ("even if the list comes back empty, still try the per-user lookup once so I can see what that response looks like") which preserves the eval signal without naming the chain.
