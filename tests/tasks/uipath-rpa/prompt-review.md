# `tests/tasks/uipath-rpa/` — prompt review

Existing test prompts vs. natural-user rewrites. Methodology in [hitl-prompts-review.html](../../hitl-prompts-review.html) and [CLAUDE.md](../../CLAUDE.md).

## Scope of this folder

`uipath-rpa` is the full Studio / Robot skill: create, edit, build, run, and debug `.xaml` and `.cs` coded workflows; UI automation through Object Repository targets; test-case authoring; legacy (.NET 4.6.1) projects via the separate `uip rpa-legacy` CLI. The six tests in this folder cover: modern coded testing scaffold (`coded_test_case`), three legacy-mode tasks (create / edit / refactor), template-aware project creation (`template_aware_create`), and one UI-automation smoke test (`uia_google_search`). Two subfolders: top-level (4 files) and `legacy/` (3 files).

## Insider markers seen in this folder

- **Exact CLI commands and flags inlined as instructions**, not as fallbacks: `uip rpa init --name "LoginTests" --location "." --template-id TestAutomationProjectTemplate`, `uip rpa-legacy create-project`, `uip rpa templates search --query "REFramework" --output json`.
- **Package machine names and pinned versions with rationale**: `UiPath.Testing.Activities` version `[25.4.1]` "newer 25.10.x releases synthesize a bootloader that breaks the build on this runner" — internal CI knowledge that no customer has.
- **Skill-internal pattern names**: "the partial-class pattern from the skill — NOT a base class, NOT a separate CodedWorkflowHooks class", "the multi-phase `uia-configure-target` / Object Repository pipeline".
- **Direct activity-class names dictated**: `NApplicationCard + NTypeInto + NClick`, `Excel Application Scope`, `Retry Scope` with `NumberOfRetries`/`RetryInterval` property names.
- **Hand-authored UI selector strings** embedded in the user request: `html app='chrome.exe' url='https://www.google.com'`, `webctrl tag='textarea' name='q'`.
- **Embedded raw `project.json` and `Main.xaml` fixture blocks** in the legacy edit/refactor prompts — over 50 lines of XML/JSON per prompt presented to the agent as "write exactly these two files".
- **Eval-harness verbs**: "Do not run debug — validation only", "Do NOT ask for approval, confirmation, or feedback", "Do NOT pause between planning and implementation", "Load the `uipath-rpa` skill and follow its workflow", "Run each command once and continue", "Use `--output json` on any uip commands you run" (appears in every file).
- **`report.json` eval-fixture schemas** with exact key lists (`hooks_pattern`, `hooks_class_declaration`, `template_strategy`, `create_project_command`, ...).
- **Sandbox / CI environment narration** in the user voice: "Studio Desktop is NOT available — skip any CLI commands that require `--use-studio`", "Helm-backed commands work as documented in the skill", "Live tenant calls may fail — that is expected".
- **Skill-rule callbacks**: "follow the skill's full workflow", "skip any CLI commands that require `--use-studio`".
- **Test-fixture project IDs**: GUIDs like `11111111-1111-1111-1111-111111111111`, `22222222-2222-2222-2222-222222222222`.

## Notes for the PR description

- **The biggest pattern in this folder is embedded fixture files in the user prompt.** Both legacy `edit` and `refactor` tests paste a 30+ line `project.json` and a fully namespaced `Main.xaml` into `initial_prompt` and tell the agent "write exactly these two files". That's a `setup` step disguised as a user message — a real customer already has those files on disk and wouldn't dictate every `xmlns:` declaration. Moving the fixture into a `setup.files` block (or a `before_task` shell step) would make these tests measure the actual refactor judgement instead of fixture-writing obedience.
- **`uia_google_search` hand-rolls selector strings in the prompt.** The UIA skill's whole premise is "NEVER hand-write selectors — use `uia-configure-target` exclusively" (Critical Rule 7 in `SKILL.md`). The test then provides three hand-written selectors in the user message and tells the agent to bypass `uia-configure-target`. If the goal is "test the placeholder-selector stub pattern" (the skill's documented offline path), the prompt should say "live UI capture isn't available here — use whatever offline path the skill supports" and not pre-seed the selectors. If the goal is "test fast-path selector authoring", that contradicts the skill's own Critical Rule and should be flagged with the skill owners.
- **`template_aware_create` is the gold standard for a decision-logic test.** It legitimately enumerates Official vs. Marketplace candidates and asks for a parseable decision — this is exactly when fixture-shaped prompts are justified. The wording could still be lightly humanized (the user is "InvoiceREF"-curious, not an eval rig), but the structural choices are correct.
- **CI-narration creep.** Every prompt in this folder leaks the CI runner into the user voice: "Studio Desktop is NOT available", "Helm-backed commands work as documented", "Live tenant calls may fail — that is expected", "Run each command once and continue", "5 fix attempts". This is symmetrical with what we saw in the HITL review and is the single highest-volume cleanup across the folder. The fix is structural: separate `initial_prompt` (what the user wants) from `environment` / `constraints` (what the harness can't provide).
- **`coded_test_case` over-specifies the hook shape.** Telling the agent "use the partial-class pattern from the skill — NOT a base class, NOT a separate `CodedWorkflowHooks` class" is the eval harness leaking its grading rubric into the prompt. A real customer would say "I want shared setup/teardown around every test case" and let the skill pick the right pattern. If the test's grading rubric needs to discriminate between the two shapes, the discrimination belongs in the grader, not in the prompt.
