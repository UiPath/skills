# `tests/tasks/uipath-solution/` — prompt review

Existing test prompts vs. natural-user rewrites. Methodology in [hitl-prompts-review.html](../../hitl-prompts-review.html) and [CLAUDE.md](../../CLAUDE.md).

## Scope of this folder

The `uipath-solution` skill has two halves: a **Design** half that turns a PDD (Process Design Document) into an SDD (Solution Design Document), and an **Operate** half that drives the `uip solution` CLI (`init / project add / resource refresh / pack / publish / deploy run / activate`). The 11 yaml tests are split accordingly: 5 design tests (one full e2e + four PDD→SDD integration variants) plus 2 smoke tests at the top level, and 5 CLI smoke tests under `operate/`. Together they cover whether the agent (a) routes a PDD into the design workflow, fills gaps correctly, and writes a template-compliant SDD, and (b) reaches for `uip solution …` for solution lifecycle ops rather than hand-rolling REST.

## Insider markers seen in this folder

- **"Use Autonomous mode — do not pause for checkpoints."** — names the skill's two-mode toggle (Autonomous vs Interactive, Rule 14, Phase 1) verbatim. Real customers do not say "use autonomous mode"; they either provide context and let the agent run, or they ask to be checked in with.
- **"For any `[SME REVIEW]` items, use the defaults and proceed."** — `[SME REVIEW]` and `[DEFAULT]` are gap-marker tokens defined in the skill (Critical Rule 4). A customer would never know that vocabulary.
- **"Apply `[DEFAULT]` and `[SME REVIEW]` markers as the skill instructs."** (`gap_detection.yaml`) — explicit call into the skill's rule catalogue.
- **"Before starting, load the `uipath-solution` skill. Read and follow its workflow steps exactly."** — names the skill's machine name. Equivalent of asking the user to read SKILL.md.
- **"Do NOT attempt to create a UiPath project or run any `uip` commands."** — eval-harness scope fence. Real users don't pre-emptively forbid the CLI; they just say what they want.
- **`smoke_file_reading.yaml`** — numbered Step 1/2/3 with explicit "Use the Write tool" / "Use the Read tool" / "Load the `uipath-solution` skill" / "Stop after Step 3" / sentinel string `QXR-9118-PURPLE` / synthetic process name `ZEBRACAKE-7472`. Pure harness fixture.
- **`smoke_skill_activation.yaml`** — "write to `phases.txt` the EXACT names of the three phases of SDD generation as the skill defines them, one per line. Use the names exactly as they appear in the skill body." A customer never asks the agent to dump a skill's internal section names.
- **Gap-filling defaults pre-supplied** (`product_selection.yaml`): "Framework: LangGraph / Tools: order_lookup / Memory: short-term only / Evaluation: trajectory similarity" — feeds the answers to `AskUserQuestion` so the harness can advance unattended.
- **"auto-registered"** (`resource_list_smoke.yaml`) — leaks the `resources/solution_folder/` auto-generation mechanic from Operate Rule 4.
- **"registered in its manifest"** (`project_list_smoke.yaml`) — "manifest" is reasonable English, but the implied target is the `.uipx` registration, which is solution-internal vocabulary.

## Notes for the PR description

- **Two recurring phrases account for almost all of the Design-half leakage:** "Use Autonomous mode — do not pause for checkpoints" and "For any [SME REVIEW] items, use the defaults and proceed." Both name skill-internal mechanics (the Autonomous/Interactive toggle from Phase 1 of the SDD Generation Guide, and the `[SME REVIEW]` / `[DEFAULT]` gap-marker vocabulary from Critical Rule 4). A single edit pass across `e2e_rpa_sdd.yaml`, `pdd_to_sdd.yaml`, `product_selection.yaml` — and a lighter edit on `gap_detection.yaml`, which legitimately needs the marker name — would clean up most of the folder.
- **The embedded PDDs themselves are excellent** — well-shaped, realistic, with vendor names (Workday, SAP FB60, Outlook), real volumes, real exception rates, and proper application/credential tables. The customer voice is right there in the PDD body; only the wrapper around it is harness-y. The fix is to keep the PDDs and humanize the framing, not to touch the PDD content.
- **`smoke_skill_activation.yaml` is the standout fixable case.** Asking the agent to dump the literal phase names from SKILL.md tests skill-content recall rather than skill activation. A "walk me through your phases" framing achieves the same grader signal without telegraphing that the answer lives in the skill body.
- **The `operate/` smokes are mostly good** — three of five are already natural and the other two only need a one-word fix ("manifest" → "in it", "auto-registered" → "out of the box"). This subfolder is a good template for what a CLI-surface smoke test should look like: name the outcome the user wants, never name the command or the flag.
- **`gap_detection.yaml` is borderline legitimate-insider.** Because the grader has to detect surviving `[SME REVIEW]` markers in the SDD, the prompt has to communicate "don't resolve gaps." That's fair. But it can do so without naming the token: "Don't ask me to fill in the blanks — leave anything you can't infer flagged for someone with business context to review later" gets the same behavior with a customer voice.
