**Do NOT misattribute:** This is **not** a bug in `HandleAppRequest` / the WorkflowEvents package, **not** a SignalR hub / `InitializeHubConnection` / `AppRequestTrigger` connection or transport failure (there is no timeout, IO, or "connection lost" / "SignalR" phrase anywhere in the evidence), and **not** an Orchestrator or licensing problem. **Immediate fix:** In `ProcessRequest.xaml`, ensure `io_Records` is assigned a non-null value before it is used, or guard the null case (validate / default the argument, e.g. initialize an empty `DataTable`) before `io_Records.Rows` is read. Correct the UiPath App's **Trigger Workflow** binding so it actually supplies the `io_Records` argument (matching name and type) when invoking the workflow.

**Preventive fix:** Treat App-invoked workflows like any callable contract — validate required input arguments at the start of the workflow and fail with a clear message when a required input is missing, rather than letting a downstream dereference fault with an opaque `NullReferenceException`. Re-verify the App's input bindings whenever the workflow's argument list changes.
