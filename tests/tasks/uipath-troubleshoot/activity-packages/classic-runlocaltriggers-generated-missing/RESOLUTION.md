# Final Resolution

Here is the investigation result.

---

**Root Cause:** The `OrderEntryAssistant` project uses the **Run Local Triggers** activity, whose
read-only trigger workflow is **auto-generated by Studio** into the hidden design-time cache at
`.local\generated\Triggers.Generated.xaml`. That generated workflow is **missing** from the package the
robot ran, so at run time `Run Local Triggers` cannot load it and the job faults ~2 seconds after start.
The Error log is **generic** — `System.Exception: The invoked workflow could not be loaded. A referenced
workflow file was not found in the running package.` at `GeneratedWorkflowLoader.Load` →
`RunLocalTriggers.Execute` — it does **not** name the file. The cause is established from the faulting
activity (`Run Local Triggers`) plus the project layout (no `.local` generated workflow) and the
knowledge that Run Local Triggers loads an auto-generated trigger workflow. This is **not** a bug in
`Main.xaml` itself — the workflow is fine; the generated artifact it depends on was never
produced/shipped.

**What went wrong:** The `.local` folder is a Studio-side design-time cache that is not normally
committed to source control. The project was built/published without a freshly generated
`Triggers.Generated.xaml` — typically because it was cloned/shared without `.local`, the cache was
stale/corrupt, or Studio and the Assistant/robot drifted to mismatched versions after an upgrade — so
the generated trigger workflow was absent at run time.

**Why (full causal chain):**
- `Main.xaml` contains a `ui:RunLocalTriggers` ("Run Local Triggers") activity. At run time this activity
  loads and starts the read-only workflow Studio generates for the project's local triggers,
  `.local\generated\Triggers.Generated.xaml`.
- That file is absent from the running package (`.local` is a non-committed design-time cache; the
  package was built without regenerating it).
- `RunLocalTriggers.Execute` → `GeneratedWorkflowLoader.Load` throws — surfaced generically as
  "the invoked workflow could not be loaded / a referenced workflow file was not found" — faulting the
  job at startup. The generic wording can be mistaken for an `Invoke Workflow File` path problem, but
  the source has no `Invoke Workflow File`; the missing workflow is the auto-generated trigger workflow.
- The job is a manually-started attended process; Orchestrator recorded the Faulted state.

**Evidence**

### classic-activities (Root Cause)
- Faulting activity: `Run Local Triggers` (`ui:RunLocalTriggers`) in `Main.xaml`.
- Error log (generic — does **not** name the file): `System.Exception: The invoked workflow could not
  be loaded. A referenced workflow file was not found in the running package.` at
  `UiPath.Studio.LocalTriggers.GeneratedWorkflowLoader.Load` → `UiPath.Core.Activities.RunLocalTriggers.Execute`.
  The `GeneratedWorkflowLoader` + `RunLocalTriggers` frames are the tell that the missing workflow is
  Run Local Triggers' auto-generated `.local\generated\Triggers.Generated.xaml`, not a hand-authored
  `Invoke Workflow File`.
- Source: `Main.xaml` is a small, valid workflow (`Log assistant ready` → `Run Local Triggers`); the
  project depends on `UiPath.Form.Activities`; there is **no `.local` folder** in the project — the
  generated trigger workflow was never produced/shipped.
- Info/Trace timeline: execution started, "starting local triggers", then `Run Local Triggers` faulted —
  the failure is at trigger startup / generated-workflow load, not while handling any trigger event.

### orchestrator (Propagation)
- Job: `OrderEntryAssistant`, folder `Operations`, job key `7e2a9c14-3b58-4d6f-a091-5c8e2f1b7d63`,
  package version `1.0.1`.
- State transition: Running `2026-07-21T14:02:11Z` → Faulted `2026-07-21T14:02:13Z` (~2s) — an immediate
  startup fault, not a mid-run failure.
- Run mode / machine: Attended, on `MOCK-HOST`. Orchestrator recorded the Faulted job.

**Immediate fix**

### classic-activities (Root Cause)
1. **Regenerate the Run Local Triggers workflow in Studio and republish.** Open (or rebuild) the project
   in UiPath Studio so it regenerates `.local\generated\Triggers.Generated.xaml`; if the cache is
   stale/corrupt, delete the `.local` folder first, then reopen so Studio rebuilds it. Republish the
   package so the generated trigger workflow ships to the robot/Assistant. **Do not** hand-create or
   hand-edit `Triggers.Generated.xaml` — it is a read-only generated artifact.
   - **Where:** the project directory (`.local\generated\`), then Studio → Publish.
   - **Who:** RPA developer.
2. **If the fault followed a Studio/Assistant (or robot) upgrade**, align their versions and republish —
   a Studio↔Assistant version mismatch produces the same missing-generated-workflow error.
3. **Re-run and confirm** the local triggers start (no `Triggers.Generated.xaml` error) and the assistant
   begins monitoring.

**Preventive fix**

1. **classic-activities** — Do not rely on the committed/cloned `.local` cache. Ensure the publish step
   runs from a Studio that has (re)generated the trigger workflow; treat `Triggers.Generated.xaml` as a
   build output, not a source artifact.
2. **orchestrator** — Add failure alerting on Faulted instances of attended trigger automations so a
   startup fault is caught rather than leaving the assistant silently not listening.
