# `tests/tasks/uipath-agents/` — prompt review

Existing test prompts vs. natural-user rewrites. Methodology in [hitl-prompts-review.html](../../hitl-prompts-review.html) and [CLAUDE.md](../../CLAUDE.md).

## Scope of this folder

The `uipath-agents` skill covers both **coded agents** (Python — LangGraph, LlamaIndex, OpenAI Agents) and **low-code agents** (Agent Builder, `agent.json`) through their full lifecycle: scaffold, init, schema sync, bindings, run, evaluate, deploy, and embed in flows / solutions. The 75 yamls split into a 43-file `lowcode/` block (mostly tool-wiring / guardrail / inline-in-flow variations) and 32 root files covering coded smoke tests, framework selection, bindings, deploy, eval, HITL, file attachments, antipatterns, and tracing. Source: `skills/uipath-agents/SKILL.md` (Project Type Detection + Task Navigation matrix).

## Insider markers seen in this folder

- **Eval-harness boilerplate trailer**: nearly every prompt ends with a 3-line *"Do NOT publish, upload, or deploy. Do NOT ask for approval, confirmation, or feedback. Do NOT pause between planning and implementation. Complete the entire task end-to-end in a single pass."* footer — pure harness, no customer would write it.
- **Framework name leakage**: prompts that say "Build a **LangGraph** UiPath coded agent" or "Build a **LlamaIndex** workflow" or "Build an **OpenAI Agents** UiPath coded agent" — naming the framework is a coverage knob, not a customer goal. Borderline acceptable when the test is specifically about that framework; insider when the goal is just "classify a ticket."
- **Embedded full source files**: `pyproject.toml`, `main.py`, `entry-points.json`, `bindings.json`, `langgraph.json`, `openai_agents.json` pasted verbatim with sentinel UUIDs (`cc111111-2222-...`, `bb111111-...`, `aaa11111-...`, `ws-baseline-entrypoint`). Several antipattern + bindings + roundtrip + workspace tests rely on this fixture-paste pattern.
- **CLI literacy**: literal `uip codedagent pack`, `uip codedagent run agent '{...}'`, `uip login`, `uip solution init`, `uip solution project add`, `uip maestro flow registry list --local --output json`, `uip maestro flow validate`, `uip solution resource list --kind App --source remote --output json`, `uip agent guardrails list/catalog`. A customer says "publish the agent" or "set up the bindings"; the harness names the executable.
- **Skill-rule callbacks**: "consult `references/coded/lifecycle/bindings-reference.md` if needed", "follow the bindings sync workflow in the uipath-agents skill", "the bindings reference allows…", "per the bindings reference". The `guardrail_recommend_all/scoped/validate` tests reference `uip agent guardrails catalog` by name.
- **Numbered step lists**: `coded_in_flow_e2e` enumerates a 6-step CLI pipeline (`1. Create solution → 2. Scaffold → 3. Register → 4. Discover → 5. Wire → 6. Validate`). `push_pull_roundtrip` numbers a 3-step edit/upload/pull sequence.
- **Sentinel test data**: `gpt-4o-mini-2024-07-18`, `temperature=0`, `ws-baseline-00000000-0000-0000-0000-000000000000`, `a54c3bf0-985c-45a7-a1c4-92e6d8888faa`, `RAN_OK` marker file, `reviewer@example.com (type 3)`, `C-101`.
- **Schema-direction & resource-kind vocabulary**: literal mentions of `entry-points.json`, `bindings.json` `metadata.SubType`, `resource.json` `isEnabled` / `properties.folderPath`, "Agent level scope", `$actionType: "escalate"`, `$guardrailType: "builtInValidator"`, `$parameterType: "enum-list"`, `selector.scopes: ["Agent"]`, `validatorType: pii_detection`. Customers don't talk in schema keys.
- **Internal pattern / framework machine names**: "Simple Function" / "Coded Function" capitalized as a framework choice, `connector key uipath-uipath-airdk`, `Tool.Guardrail.Escalation.Action.App`, `LLMJudgeOutputEvaluator`, `LLMJudgeTrajectoryEvaluator`.
- **Antipattern trap markers**: prompts explicitly tell the agent "**Recreate this code exactly**, then review and fix anti-patterns in place" — necessary mechanic for an antipattern test, but the framing is unmistakably harness-shaped.
- **"Use the real folder/process/index name from the tenant — do not hard-code"**: a fixture-discovery instruction repeated verbatim across ~14 external/inline-external low-code tests. Customers don't say "use the real folder from my tenant" — they just say the folder name.

## Notes for the PR description

- **The "do not pause, do not ask for approval" footer is universal harness boilerplate** that doesn't belong in customer voice. It appears in roughly 65 of 75 prompts. Recommend moving it to a global eval-runner config or system prompt rather than baking it into every initial_prompt.
- **Framework-name leakage is the most consistent fixable pattern.** ~12 prompts say "Build a LangGraph / LlamaIndex / OpenAI Agents UiPath coded agent" up front. If framework selection is the test's measurement, that's a fair stipulation; if the test is about classification / RAG / handoff capability, the framework name should *not* leak — the agent should pick it. Confirm with maintainers whether each framework smoke test is measuring framework selection or capability.
- **The "Use the real folder and process name from the tenant — do not hard-code or guess" copy is repeated verbatim across ~14 external-tool / inline-external-tool low-code tests.** This is a fixture-discovery affordance for the eval; a real customer either names the folder or doesn't think to mention it. Recommend either dropping the directive (let the test grade folder lookup as a sub-skill) or rewording as "look it up if you need to."
- **Low-code structurally identical pairs** (`external_X` / `inline_external_X` / `solution_X` / `inline_solution_X` matrices) are highly templated — when one is fixed, its three siblings need the same fix. The solution-internal variants (`solution_*` and `inline_solution_*`) read more naturally than the external variants because they don't carry the "Use the real folder from the tenant" affordance.
- **Antipattern / fixture-paste tests should arguably move out of `initial_prompt`** into a separate `seed_files` block in the YAML. Right now the user-message has to contain full multi-file Python/TOML/JSON dumps to set up the fixture, which structurally can't be naturalized. Splitting fixture-seed from user-intent would let those tests show clean prompts like "I'm porting a legacy LangGraph classifier into UiPath — can you review what I've got and fix anything that won't work?" while the fixture lives elsewhere.
- **Standout naturals worth modeling:** `guardrail_recommend_all`, `guardrail_recommend_scoped`, `hitl_create_task_with_app`, `file_attachment_input`, `deploy_folder_and_rebump`, `init_validate`. These prompts read like a real engineer dropped them into a chat window — short, business-first, no internal vocabulary. Good template targets when rewriting the fixable bucket.
- **Standout offenders worth surfacing:** `coded_in_flow_e2e` (6-step CLI enumeration), `openai_agents_handoff` (Pydantic class names + `Agent[...]` syntax + `openai_agents.json` key in customer voice), `bindings_sync` (5-step "the bindings skill should…" directive list), `tracing_redaction` (less bad but explicitly names spans / labels in user voice).
