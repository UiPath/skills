# `tests/tasks/uipath-data-fabric/` — prompt review

Existing test prompts vs. natural-user rewrites. Methodology in [hitl-prompts-review.html](../../hitl-prompts-review.html) and [CLAUDE.md](../../CLAUDE.md).

## Scope of this folder

The `uipath-data-fabric` skill wraps the `uip df` CLI for UiPath Data Fabric — UiPath's structured data store. It covers entity schema CRUD (create/get/update — never delete), record CRUD (insert/get/update/query/delete, both single and batch), CSV bulk import, file-attachment upload/download/delete, complex filter/sort/group-by queries, and cursor + offset pagination. The 15 yaml tests in this folder are split across `smoke_*` (offline CLI/refusal coverage with placeholder IDs), `integration_*` (live-tenant CRUD/pagination/files/error-paths), and `e2e_*` (realistic business scenarios). Entity/field/record vocabulary itself is customer-facing (these are Data Fabric concepts), but most prompts go further and dictate the exact CLI commands, flag combinations, and step-by-step verification scripts.

## Insider markers seen in this folder

- **Boilerplate harness preamble on nearly every prompt**: "Before starting, load the uipath-data-fabric skill and follow its workflow."
- **CLI flags named in user voice**: `--output json` (on essentially every prompt as a "do this on every call" constraint), `--native-only`, `--cursor`, `--limit`, `--offset`, `--file`, `--destination`, `--body`, `--folder`, `--user`.
- **CLI gotcha trivia stated as customer-side constraint**: "`--offset` and `--cursor` are mutually exclusive — pick one per command, never both", "the download flag is `--destination` (not `--output`)", "the payload key is `id` (lowercase), not `ID`", "CSV headers must match entity field names exactly — case-sensitive".
- **Eval-harness verbs**: "Run each command once and move on", "Do NOT retry", "Do NOT re-login", "Do NOT call the DataFabric REST API", "MANDATORY — this step MUST be completed in full before any later step", "Run each page as its OWN Bash invocation … not via Python subprocess", "Save a summary to report.json", "if step 5 did not actually produce 9 separate tool calls, return to step 5 and finish the walk".
- **Internal JSON shape leakage**: `filterGroup`, `logicalOperator: 0 (AND) / 1 (OR)`, `queryFilters`, `selectedFields`, `sortOptions`, `addFields`, `updateFields`, `removeFields`, `Fields[].ID`, `NextCursor`, `HasNextPage`, `InsertedRecords`, `TotalRecords`, exact `EntityFieldDataType` literals (`STRING` / `INTEGER` / `DECIMAL` / `BOOLEAN` / `DATE` / `MULTILINE_TEXT` / `FILE`), constraint key names (`lengthLimit` / `minValue` / `maxValue` / `decimalPrecision` / `isRequired` / `isUnique`).
- **Embedded full JSON record bodies and CSV blocks** in the prompt (notably the inlined 20-row Python `csv` script in `integration_preseeded_entity`).
- **Test-fixture leakage**: placeholder entity IDs (`ent-456-xyz`, `task-001-ent`, `doc-001-ent`, `prod-002-ent`, `old-ent-111`, `cust-222-ent`, `sf-333-ent`, `rec-001-id`, `id-single`, `id-2`, `id-3`, `cursor-page-2`, `seed_id`, `seed_id2`, `field-hire-id`, `field-deptname-id`); fake emails `user1@test.com`..`user20@test.com`; the literal zero-GUID `00000000-0000-0000-0000-000000000000`.
- **Skill-rule callbacks**: "follow the skill's workflow" appears in nearly every prompt; `smoke_negative_guards` is built around the skill's own *Not Supported* table.
- **Numbered step lists ranging from 3 to 17 sequential CLI invocations** dictated to the agent (peak: `integration_error_paths` at 17 steps).
- **Expected-result oracles embedded in the prompt** ("Expected: exactly 1 result (ORD-002)", "Expected: 9 pages", "records[0].Score == 10.0", "12 records") — these are graders, not user voice.

## Notes for the PR description

- **The folder is dominated by legitimate-insider tests (9 of 15) — a useful finding by itself.** Five offline `smoke_*` tests need placeholder IDs and CLI-flag literacy because there's no live tenant to introspect. Three `integration_*` tests (`error_paths`, `large_pagination`, `query_filters`) are coverage matrices whose grading depends on naming the specific API mechanics. `integration_preseeded_entity` is a known-answer query oracle whose 20-row fixture is the grader. None of these should be rewritten in customer voice without first redesigning the eval harness.
- **The five "fixable" prompts cluster on a single pattern**: a believable business goal at the top, then a 6–12 step numbered CLI script underneath. The rewrites strip the numbered scripts and keep the goal — this is the cleanest place to recover "is the skill self-describing enough that an agent can sequence these CLI calls itself?" The two e2e tests (`employee_directory`, `product_catalogue`) are the closest to customer voice already and only need the harness preamble + `Do NOT delete the entity` + type-name slips removed.
- **One concrete recommendation for maintainers**: `integration_preseeded_entity` is the strongest candidate for moving its seed CSV and expected-count oracle out of `initial_prompt` and into a sibling fixture file referenced by the eval harness. The test's purpose (filter-grammar correctness against known answers) doesn't require those answers to be visible to the agent — and exposing them in the prompt arguably leaks the grader.
- **Recurring "Use `--output json` on every uip df command" line**: this constraint appears in 11 of 15 prompts. If the skill's CLI default were `--output json` (or if the eval harness set it via env var), this entire phrase could disappear from the user-voice prompts and the tests would be unambiguously stronger.
- **`task_id` naming is consistent and good** (`skill-datafabric-{smoke,integration,e2e}-<scenario>`). No change needed.
