Resolution Engine

How a metric name becomes a TypeScript widget — with examples

Entry point — intent.json

→
The agent writes only a metric name + tier. The build script does everything else.

Tier 2
Parametric — Typed Descriptor Compiled to TypeScript
Known SDK service + user-supplied filter values · agent writes params only, not code
⚡ <2s / widget
Step-by-step — "queues where failure > 20"
// Agent writes in intent.json:
{
  "name": "queue-failure-threshold",
  "tier": "T2",
  "params": {
    "threshold": 20,
    "direction": "gt"
  }
}
1 Registry lookup → Queues service, getAll(), filterField: "failureCount"
↓
2 JSON schema validation of params (op must be gt/lt/eq/gte/lte/neq)
↓
3 compileT2ToTypeScript(descriptor) → safe TS function body
↓
✓ Inject into sdk-data-table template — agent wrote zero TypeScript
What compileT2ToTypeScript produces
// Build script compiles this from params — never agent-authored:
const svc = new Queues(sdk as never)
const result = await svc.getAll({})
const items = (result?.items ?? [])
  as Array<Record<string, number>>

// "gt" → ">" · threshold=20 · filterField="failureCount"
const filtered = items.filter(
  item => (item.failureCount ?? 0) > 20
)
items.sort((a, b) => b.failureCount - a.failureCount)
return filtered
The agent only provided threshold: 20 and direction: "gt".
The build script derived: import path, class name, method, field name, JS operator, sort order.
↓
Tier 3
Custom — Two Sub-paths for Novel Metrics
T3-Insights: useInsights with a valid InsightsKey · T3-SDK: typed fnBody with tsc enforcement
⏱ 5–10s + retry
Sub-path A — T3-Insights (validated InsightsKey, standard template)
// Agent writes:
{
  "name": "incident-distribution",
  "tier": "T3",
  "title": "Incident Types",
  "namespace": "agents",
  "method": "getIncidentDistribution",
  "template": "donut-chart",
  "dataSelector": "(data as any)?.data ?? []"
}
1 Check "agents.getIncidentDistribution" is in VALID_INSIGHTS_KEYS (28 keys)
→ ✓ valid — build continues. If invalid, fails immediately with the valid list.
↓
2 Build script derives useInsights('agents.getIncidentDistribution', ...) hook
↓
✓ Applies to donut-chart template — same path as T1, no agent TypeScript
Sub-path B — T3-SDK (agent writes async fn body, tsc enforces the interface)
// Agent writes fnBody (just the data logic):
const { Jobs } = await import(
  '@uipath/uipath-typescript/jobs'
)
const svc = new Jobs(sdk as never)
const r = await svc.getAll({ state: 'Running' })
return (r?.items ?? []).map(
  (j: any) => ({
    name: j.processName,
    state: j.state
  })
)
1 Build script injects fnBody into t3-shell.tsx.template
→ Component shell (card, loading, error boundary) is always deterministic
↓
2 tsc --noEmit on the generated file
→ TypeScript enforces: fn must return Promise<Row[]>, no JSX, valid SDK calls
↓
✗ If tsc fails — emits T3_RETRY with the errors. Agent fixes fnBody. Max 1 retry, then exit 2.
↓
✓ If tsc passes — widget written. WIDGET_READY emitted.
↓
Hard Refuse
Checked before any tier routing
6 impossible requests · agent refuses only the metric, builds dashboard without it
❌ REFUSED
"agent cost in dollars"
"CPU per agent"
"who triggered this job"
"cross-tenant comparison"
"SLA breach %"
"error stack traces"
WHY NOT POSSIBLE
Platform tracks AGU, not $
Not exposed by any API
No end-user identity in job records
Single-tenant scope only
No SLA metadata in platform
No aggregation endpoint
OFFERED INSTEAD
invocation-volume (AGU chart)
agent-latency (fleet-level)
job-completion-trend by process
multi-widget single-tenant view
success rate % (computable)
agent-errors (count by type)
T1 KEY PROPERTY
Agent writes the metric name. Build script writes all TypeScript. Zero hallucination risk.
T2 KEY PROPERTY
Agent writes values (threshold, direction). Build script compiles to TypeScript. No code generation by AI.
T3 KEY PROPERTY
Agent writes only a function body. tsc enforces the interface. Component shell is always deterministic.
