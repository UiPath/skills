2
Build Time — build-dashboard.mjs generating widget code
Template substitution, T2 compilation, T3 injection, tsc validation
Current — unvalidated string literals
OK T3-SDK: tsc --noEmit catches type errors in fnBody
OK T3-Insights: VALID_INSIGHTS_KEYS validates method before codegen
GAP T1 templates: yKey="value" is a string — tsc doesn't know "value" must exist on the API response
GAP T2 filterField: "failureCount" is unvalidated — can compile and silently return 0 rows
GAP viewColumns keys: string literals — TypeScript has no way to validate them against response shape
Right approach — typed interfaces per metric
// Today — no type safety on field names:
const chartData: Record<string, unknown>[] =
  (data as any)?.data ?? []

// With SDK types — field errors caught at tsc:
import type { AgentErrorItem } from
  '@uipath/uipath-typescript/insights'
const chartData: AgentErrorItem[] =
  response?.data ?? []
// chartData[0].value ✓ · .wrongField ✗ compile error
When Insights is in the SDK, every widget is strongly typed end-to-end. Wrong field names = tsc error before the user ever runs the dashboard.
3
Runtime — browser executing the dashboard
Last line of defence — should never be the only line of defence
✓ Already in place
autoColumns(rows) in detail views — detects real field names from first data row at runtime. If viewColumns is wrong, the view still shows actual data.
✗ Still a gap
Main widgets (KPI cards, charts) use template-substituted keys like yKey="value". If "value" doesn't exist, the chart renders empty with no error shown to the user.
→ Right fix
Strongly typed widget templates (possible when Insights is in SDK). tsc catches wrong field names before runtime. Runtime validation becomes redundant, not primary.
Closing the gap — ordered by impact
1
Auto-generate sdk-field-reference.md from SDK release artefacts
A script runs on SDK version bump → regenerates the file → no manual curation. Agent's context is always the real SDK types.

2
Validate T1/T2 registry field names at build-script startup
Build script reads SDK type definitions and verifies each registry entry's filterField, yKey, viewColumns keys exist on the declared response type. Fails loudly at build time.

3
When Insights SDK ships — replace useInsights + InsightsClient with typed SDK service calls
(data as any)?.data ?? [] becomes response.data ?? [] typed against SDK response interfaces. Wrong field access = compile error. The tier system has one data source.
When SDK ships
4
Remove all as any casts — widget templates typed end-to-end against SDK interfaces
T3-SDK fnBody already validated by tsc. T1/T2 generated code validated by tsc. Wrong field names are compile errors, not silent — values.
