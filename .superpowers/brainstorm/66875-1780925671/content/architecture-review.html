Architecture Review — Dashboard Skill

Is the current path right? What would we do differently?

✅ What's working well

Code-as-Orchestrator principle
The core insight — agent writes a minimal intent.json, build script generates all TypeScript — is sound. Agent hallucination risk is contained to metric names and params, not widget code. This is the right design.
T3-SDK tsc validation loop
Agent writes a function body, tsc --noEmit validates it, exit code 2 triggers a retry. This is a clean, tight feedback loop. The only place AI-generated TypeScript can be written, and it's enforced by the compiler.
Structured event streaming
WIDGET_READY, T3_RETRY, BUILD_RESULT — each event is a single contract between the build script and the agent. Clean, testable, and easy to extend.
Live SDK docs as source of truth
Replacing hand-maintained sdk-field-reference.md with a WebFetch of the official llms-full-content.txt was the right call. Agent always has current field names from the actual SDK, not our snapshot.

🔴 Root problems — not symptoms

We've fixed many symptoms (wrong field names, blank views, TypeScript errors) but four root problems remain:

Root Problem 1 — Everything is as any in T1/T2 widgets
The issue
T1 and T2 generated widgets use (data as any)?.data ?? []. TypeScript compiles fine regardless of what field names you put in yKey, columns, or dataSelector. Wrong field names silently render —. The bugs we've been fixing (name vs processName, value vs count, etc.) are all downstream of this.
Why it exists
The Insights RTM HTTP client returns untyped JSON. Until Insights is in the TypeScript SDK with proper response interfaces, there's no static type to validate against. as any is the honest reflection of that gap.
When it goes away
When Insights lands in the SDK. Response types become concrete interfaces. (data as any) becomes response: AgentErrorsResponse. Wrong field access = compile error. The autoColumns() workaround becomes unnecessary.
Root Problem 2 — npm ci is the bottleneck and there's no avoiding it per-build
The issue
We generate a full Vite + React project per dashboard. npm ci runs once per project (cold: 2–5 min, warm: 30s). The pre-warm mitigates this for the confirm→build step, but first runs are always slow. Users feel it.
Why it exists
We chose code generation over config-driven rendering. A compiled React app per dashboard is powerful but expensive to bootstrap.
Mitigation available now
A global npm cache shared across all dashboard projects reduces warm installs to 15–30s. Set npm ci --prefer-offline --cache ~/.uipath-dashboard-cache — packages downloaded once, reused for every dashboard.
Root Problem 3 — T2 is a thin tier with weak guarantees
The issue
T2 has 1 entry (jobs-duration-threshold). The original queue-failure-threshold entry was removed because QueueGetResponse doesn't expose failure counts. T2 was designed for parametric SDK filtering but the SDK response shapes don't always expose what users want to filter on. The filterField concept only works if the field actually exists on the response type.
Assessment
T2 is a good idea when it works. But one entry makes it look like a premature abstraction. When Insights SDK ships with richer response types, T2 can expand meaningfully. For now, it's correct but thin.
Root Problem 4 — OAuth setup is a hard blocker for non-technical users
The issue
Every dashboard requires an external OAuth app with specific scopes. Creating one requires uip admin permissions, scope discovery, and explicit Claude Code approval. If the user doesn't have admin rights, they're blocked. If the command fails, they have to create the app manually.
Why it exists
Dashboards run on localhost:57173. PKCE OAuth requires a registered redirect URI. The platform won't issue tokens to unregistered apps. This is a security requirement, not a skill limitation.
Better path
One OAuth app per tenant, created once by an admin, shared across all dashboards. The clientId is stored in the user's environment or retrieved from a tenant-level config. Not re-created per dashboard.

🔷 Is there a fundamentally better approach?

Option A — Config-driven runtime rendering (no per-dashboard compilation)
Instead of generating TypeScript files per dashboard, generate a JSON dashboard config. A single pre-built, pre-compiled React app reads the JSON and renders widgets.

✓ Eliminates npm ci entirely — first dashboard in ~15s
✓ No tsc per dashboard — compile happens once for the shared app
✓ The JSON config is strongly typed and easy to validate
✓ Agent writes JSON params, not TypeScript — no hallucination in code
✗ Loses T3-SDK flexibility — can't inject arbitrary TypeScript functions into JSON
✗ Significant rewrite — shared app needs all 14 widget types parameterised
✗ Custom widgets require the shared app to be re-deployed

Best for: The primary user persona (Admin/CoE Lead) who needs T1/T2 metrics. T3-SDK is a power-user feature they probably don't need. If 90% of dashboards are T1-only, this is the right trade.
Option B — Keep current approach, but address root problems now
Stay with code generation but make targeted fixes:

✓ Now: Shared npm cache (--cache ~/.uipath-dashboard-cache) — warm installs in 15s
✓ Now: One OAuth app per tenant (admin creates it once, stored as env var) — remove per-dashboard OAuth flow
✓ Now: Replace <PLACEHOLDER> template substitution with typed component factories — no silent failures
✓ When SDK ships: Replace as any with proper response interfaces — TypeScript catches wrong fields
✓ When SDK ships: T2 expands to include Insights parametric queries

✗ Still slower than config-driven (5–30s warm, 2–5min cold)
✗ Still requires TypeScript compilation per dashboard

Best for: Preserving T3-SDK flexibility while improving on the current pain points incrementally.
Option C — Hybrid (config-driven for T1/T2, code-gen for T3)
Run two pipelines based on dashboard composition:

✓ If all metrics are T1/T2: render from JSON config in pre-built app → 15s
✓ If any metric is T3-SDK: fall back to full code generation → 3-5min
✓ Best of both worlds for the common case
✗ Two build paths = twice the code to maintain
✗ Config-driven app needs to be maintained alongside widget templates

Best for: Optimising the common case (most dashboards are T1-only) while not losing T3 power.

📌 Recommendation

The architecture is on the right path. The pain points are real but solvable.

The Code-as-Orchestrator principle, tiered resolution, and event streaming are all correct decisions. The problems are in the implementation layer, not the architecture itself.

The two highest-impact changes to make now:

1
Shared npm cache — eliminates the cold-start problem now
Set npm ci --prefer-offline --cache ~/.uipath-dashboard-cache in runPrewarm(). First install populates the cache. Every subsequent dashboard reuses it. Cold start goes from 3–5min to 30s.
2
Tenant-level OAuth app — removes the per-dashboard OAuth flow
One external app per tenant, created by an admin, stored in env or user config. The build script reads it automatically. Phase 4 (OAuth client) disappears from the flow entirely for returning users.

The as any problem and full config-driven rendering are both solved when Insights lands in the SDK. The architecture is already structured to accommodate that — useInsights is isolated in one file, T3-Insights maps directly to future SDK methods. The migration will be a targeted swap, not a rewrite.

Longer-term consideration
Option C (hybrid) is worth exploring after the Insights SDK ships. At that point, T1/T2 metrics can move to a config-driven renderer (no compilation, 15s builds) while T3-SDK retains the code generation path for advanced use cases. The current architecture is the right foundation for that hybrid — the tier separation already exists.
